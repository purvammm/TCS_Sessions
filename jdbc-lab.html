🔄 Derby giving you trouble? Use H2 instead — every lab works identically

H2 is a tiny, zero-setup Type 4 database. Add the JAR/dependency and change just three things — the rest of the code is unchanged:

// 1. Driver class (Class.forName is optional in JDBC 4.0+):
org.h2.Driver

// 2. Connection URL (file-based, auto-creates):
jdbc:h2:~/employeedb

// 3. Maven dependency:
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <version>2.2.224</version>
</dependency>

User sa, blank password — same as Derby. The DDL, INSERT, SELECT, UPDATE, DELETE code is all identical.

LAB 1▶ Your First JDBC Connection
Concept All five JDBC steps in their simplest form — open a connection, confirm it's live, close it. This is the tactile anchor before CRUD begins.
  1. Create package com.tcs.jdbc, class JDBCTest.java.
  2. Add the code below and run it.
package com.tcs.jdbc;

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class JDBCTest {
    public static void main(String[] args) {
        try {
            // Step 1: Register the driver (optional in JDBC 4.0+, shown for clarity)
            Class.forName("org.apache.derby.jdbc.EmbeddedDriver");

            // Step 2: Establish the connection
            String url = "jdbc:derby:employeedb;create=true";
            Connection con = DriverManager.getConnection(url, "sa", "");

            // Step 3-4: Verify the connection is open
            System.out.println("Connected: " + !con.isClosed());

            // Step 5: Close it
            con.close();
            System.out.println("Closed: " + con.isClosed());

        } catch (ClassNotFoundException e) {
            System.out.println("Driver not found: " + e.getMessage());
        } catch (SQLException e) {
            System.out.println("DB error: " + e.getMessage());
        }
    }
}
Expected output
Connected: true Closed: true
SymptomCauseFix
ClassNotFoundExceptionderby.jar isn't on the classpath.Add the Derby JAR to the build path (Lab 0).
Path-with-spaces errorBackslashes / spaces in a Windows path.Use forward slashes in the URL, e.g. jdbc:derby:C:/data/employeedb;create=true.

Heads-up Running this twice throws "table already exists" (SQLState X0Y32/42Y55). That's fine — either wrap it in a try-catch, or drop the table first if you're re-running. Then verify: in Data Source Explorer, refresh the connection → expand Tables → EMPLOYEES appears.

Part B — INSERT with PreparedStatement

public static void insertEmployee(Connection con, int id, String first,
                                  String last, String dept, double salary)
                                  throws SQLException {
    String sql = "INSERT INTO employees VALUES (?, ?, ?, ?, ?)";
    PreparedStatement ps = con.prepareStatement(sql);

    ps.setInt(1, id);          // parameter index starts at 1, not 0
    ps.setString(2, first);
    ps.setString(3, last);
    ps.setString(4, dept);
    ps.setDouble(5, salary);

    int rows = ps.executeUpdate();
    System.out.println(rows + " row(s) inserted.");
}

Call it three times with different data:

insertEmployee(con, 101, "Ravi",  "Sharma", "Engineering", 75000.00);
insertEmployee(con, 102, "Priya", "Mehta",  "HR",          65000.00);
insertEmployee(con, 103, "Arjun", "Patel",  "Engineering", 80000.00);
Expected output
1 row(s) inserted. 1 row(s) inserted. 1 row(s) inserted.
Verify in Data Source Explorer: right-click EMPLOYEES → Data → Edit — all 3 rows visible.
Observation — try breaking it on purpose What happens if you insert emp_id 101 again? Run it and read the error carefully — note the SQLState code. (It's 23505 — duplicate primary key. We'll handle this gracefully in Lab 5.)
LAB 3SELECT▶ SELECT All — Reading a ResultSet
Concept A SELECT returns a ResultSet — a cursor over the rows. rs.next() moves forward one row and returns true; the while loop stops when it returns false. 
String sql = "SELECT * FROM employees";
Statement stmt = con.createStatement();
ResultSet rs = stmt.executeQuery(sql);

System.out.println("ID  | Name            | Department  | Salary");
System.out.println("----+-----------------+-------------+--------");

while (rs.next()) {
    System.out.printf("%-4d| %-16s| %-12s| %.2f%n",
        rs.getInt("emp_id"),
        rs.getString("first_name") + " " + rs.getString("last_name"),
        rs.getString("department"),
        rs.getDouble("salary"));
}
Expected output
ID | Name | Department | Salary ----+-----------------+-------------+-------- 101 | Ravi Sharma | Engineering | 75000.00 102 | Priya Mehta | HR | 65000.00 103 | Arjun Patel | Engineering | 80000.00
How the cursor works rs.next() starts before row 1. Each call advances one row and returns true while rows remain, false when they're exhausted — so the while loop ends naturally. You read columns by name (rs.getString("first_name")) or by 1-based index.
Try it yourself Write getEmployeesByDepartment(String dept) that returns only Engineering employees. Hint: use WHERE department = ? with a PreparedStatement and bind the value with ps.setString(1, dept).
LAB 4SELECTUPDATEDELETE▶ SELECT by ID, UPDATE, DELETE

SELECT by ID — use if(rs.next()), not while

Why if here? We expect at most one row (the id is the primary key), so a single if(rs.next()) is clearer than a loop — and it gives us a clean "not found" branch. 
String sql = "SELECT * FROM employees WHERE emp_id = ?";
PreparedStatement ps = con.prepareStatement(sql);
ps.setInt(1, empId);
ResultSet rs = ps.executeQuery();

if (rs.next()) {
    System.out.println("Found: " + rs.getString("first_name") +
                       " " + rs.getString("last_name"));
} else {
    System.out.println("No employee found with ID: " + empId);
}
Expected — test both cases
empId = 101 -> Found: Ravi Sharma empId = 999 -> No employee found with ID: 999

UPDATE

String sql = "UPDATE employees SET salary = ? WHERE emp_id = ?";
PreparedStatement ps = con.prepareStatement(sql);
ps.setDouble(1, 90000.00);
ps.setInt(2, 101);

int rows = ps.executeUpdate();
System.out.println(rows > 0 ? "Salary updated." : "Employee not found.");

DELETE

String sql = "DELETE FROM employees WHERE emp_id = ?";
PreparedStatement ps = con.prepareStatement(sql);
ps.setInt(1, 103);

int rows = ps.executeUpdate();
System.out.println(rows > 0 ? "Deleted." : "Employee not found.");
Run SELECT all after each operation Confirm 101's salary changed to 90000.00 and 103 is gone. Re-running SELECT all between operations is the fastest way to see your changes land.
Why the rows-affected check matters executeUpdate() returns the number of rows changed. Checking rows > 0 lets you distinguish "query ran fine but matched nothing" from "record found and modified" — far more useful than assuming success.
LAB 5▶ try-with-resources Refactor
The problem we're fixing Every lab so far has a risk: if an exception fires between getConnection() and con.close(), the connection leaks. try-with-resources auto-closes every resource — on success and on failure — eliminating the leak. 
public static void insertEmployee(int id, String first, String last,
                                  String dept, double salary) {
    String url = "jdbc:derby:employeedb;create=true";
    String sql = "INSERT INTO employees VALUES (?, ?, ?, ?, ?)";

    try (Connection con = DriverManager.getConnection(url, "sa", "");
         PreparedStatement ps = con.prepareStatement(sql)) {

        ps.setInt(1, id);
        ps.setString(2, first);
        ps.setString(3, last);
        ps.setString(4, dept);
        ps.setDouble(5, salary);
        ps.executeUpdate();
        System.out.println("Inserted successfully.");

    } catch (SQLException e) {
        System.out.println("Error    : " + e.getMessage());
        System.out.println("SQLState : " + e.getSQLState());
        System.out.println("ErrorCode: " + e.getErrorCode());
    }
}
Deliberately trigger a failure Call this with a duplicate emp_id and watch all three SQLException fields print:
Error : The statement was aborted because it would have caused a duplicate key value in a unique or primary key constraint... SQLState : 23505 ErrorCode: 20000
In a real app you'd log these with a logger — but this is exactly the information your DevOps team needs to diagnose a production DB issue.
Refactor task Update your SELECT, UPDATE and DELETE methods to also use try-with-resources. For SELECT, declare the ResultSet in the try(...) header too — it's AutoCloseable as well.
LAB 6▶ DAO Pattern — Full Refactor
The architecture Three classes, three responsibilities: DBConnector (just the connection), EmployeeDAO (all SQL), EmployeeTester (the menu & I/O). No layer does another's job.

1 · DBConnector.java

package com.tcs.jdbc;

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class DBConnector {
    private static final String URL = "jdbc:derby:employeedb;create=true";
    private static final String USER = "sa";
    private static final String PASSWORD = "";

    public static Connection getConnection() throws SQLException {
        return DriverManager.getConnection(URL, USER, PASSWORD);
    }
}

2 · EmployeeDAO.java — all five operations, the biggest block

package com.tcs.jdbc;

import java.sql.Connection;
import java.sql.PreparedStatement;
import java.sql.ResultSet;
import java.sql.SQLException;

public class EmployeeDAO {

    // ===== CREATE =====
    public void insert(int id, String first, String last,
                       String dept, double salary) {
        String sql = "INSERT INTO employees VALUES (?, ?, ?, ?, ?)";
        try (Connection con = DBConnector.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setInt(1, id);
            ps.setString(2, first);
            ps.setString(3, last);
            ps.setString(4, dept);
            ps.setDouble(5, salary);
            int rows = ps.executeUpdate();
            System.out.println(rows + " row(s) inserted.");
        } catch (SQLException e) {
            System.out.println("Insert failed [" + e.getSQLState() + "]: "
                               + e.getMessage());
        }
    }

    // ===== READ ALL =====
    public void getAll() {
        String sql = "SELECT * FROM employees";
        try (Connection con = DBConnector.getConnection();
             PreparedStatement ps = con.prepareStatement(sql);
             ResultSet rs = ps.executeQuery()) {
            System.out.println("ID  | Name            | Department  | Salary");
            System.out.println("----+-----------------+-------------+--------");
            while (rs.next()) {
                System.out.printf("%-4d| %-16s| %-12s| %.2f%n",
                    rs.getInt("emp_id"),
                    rs.getString("first_name") + " " + rs.getString("last_name"),
                    rs.getString("department"),
                    rs.getDouble("salary"));
            }
        } catch (SQLException e) {
            System.out.println("Read failed: " + e.getMessage());
        }
    }

    // ===== READ BY ID =====
    public void getById(int id) {
        String sql = "SELECT * FROM employees WHERE emp_id = ?";
        try (Connection con = DBConnector.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setInt(1, id);
            try (ResultSet rs = ps.executeQuery()) {
                if (rs.next()) {
                    System.out.println("Found: " + rs.getString("first_name")
                        + " " + rs.getString("last_name")
                        + " (" + rs.getString("department") + ")");
                } else {
                    System.out.println("No employee found with ID: " + id);
                }
            }
        } catch (SQLException e) {
            System.out.println("Read failed: " + e.getMessage());
        }
    }

    // ===== UPDATE =====
    public void updateSalary(int id, double salary) {
        String sql = "UPDATE employees SET salary = ? WHERE emp_id = ?";
        try (Connection con = DBConnector.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setDouble(1, salary);
            ps.setInt(2, id);
            int rows = ps.executeUpdate();
            System.out.println(rows > 0 ? "Salary updated." : "Employee not found.");
        } catch (SQLException e) {
            System.out.println("Update failed: " + e.getMessage());
        }
    }

    // ===== DELETE =====
    public void delete(int id) {
        String sql = "DELETE FROM employees WHERE emp_id = ?";
        try (Connection con = DBConnector.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setInt(1, id);
            int rows = ps.executeUpdate();
            System.out.println(rows > 0 ? "Deleted." : "Employee not found.");
        } catch (SQLException e) {
            System.out.println("Delete failed: " + e.getMessage());
        }
    }
}

3 · EmployeeTester.java — Scanner menu

package com.tcs.jdbc;

import java.util.Scanner;

public class EmployeeTester {
    public static void main(String[] args) {
        EmployeeDAO dao = new EmployeeDAO();
        Scanner sc = new Scanner(System.in);

        while (true) {
            System.out.println("\n==== Employee Menu ====");
            System.out.println("1 - Add Employee");
            System.out.println("2 - View All Employees");
            System.out.println("3 - Search by ID");
            System.out.println("4 - Update Salary");
            System.out.println("5 - Delete Employee");
            System.out.println("0 - Exit");
            System.out.print("Choose: ");
            int choice = sc.nextInt();

            switch (choice) {
                case 1:
                    System.out.print("ID: ");        int id = sc.nextInt();
                    System.out.print("First name: "); String f = sc.next();
                    System.out.print("Last name: ");  String l = sc.next();
                    System.out.print("Department: "); String d = sc.next();
                    System.out.print("Salary: ");      double s = sc.nextDouble();
                    dao.insert(id, f, l, d, s);
                    break;
                case 2:
                    dao.getAll();
                    break;
                case 3:
                    System.out.print("ID to search: ");
                    dao.getById(sc.nextInt());
                    break;
                case 4:
                    System.out.print("ID: ");          int uid = sc.nextInt();
                    System.out.print("New salary: ");  double ns = sc.nextDouble();
                    dao.updateSalary(uid, ns);
                    break;
                case 5:
                    System.out.print("ID to delete: ");
                    dao.delete(sc.nextInt());
                    break;
                case 0:
                    System.out.println("Goodbye!");
                    sc.close();
                    return;
                default:
                    System.out.println("Invalid choice. Try again.");
            }
        }
    }
}
Expected Run EmployeeTester. You now interact with your database through the menu — add, view, search, update, delete, all looping until you enter 0.
Observation question If you had to switch this project from Derby to MySQL, which files would change, and which would stay exactly the same?

Answer: only DBConnector (its URL + driver, and maybe minor SQL syntax in the DAO). EmployeeTester is completely untouched — the UI never knew which database it was talking to. That separation is the whole point of the DAO pattern.

Quick reference card

Interfaces & classes

Interface / ClassPurpose
DriverManagerManages drivers, returns a Connection from a URL
ConnectionOpen DB session — one socket, always close it
StatementStatic SQL — never use for user input
PreparedStatementParameterized SQL — always use this
ResultSetCursor over SELECT results, iterate with rs.next()
SQLExceptionChecked exception — getMessage, getSQLState, getErrorCode

The five steps

  1. Class.forName(driverClass) — register (auto in JDBC 4.0+)
  2. DriverManager.getConnection(url, user, pass) — connect
  3. con.prepareStatement(sql) — prepare
  4. ps.executeQuery() or ps.executeUpdate() — execute
  5. con.close() — release (or use try-with-resources)
