Session 2 recap

Let's warm up the brain. Read each question, decide your answer in your head, then click the card to reveal it. If all three feel obvious, you're ready for today.

Q1 · What does @RequestBody do? click to reveal ▸
It tells Spring to take the JSON in the request body and deserialize it into a Java object (via Jackson). You put it on a method parameter — e.g. addPost(@RequestBody PostDTO dto) — and Spring hands you a ready-built object filled with the client's data.
Q2 · What's the difference between POST and PUT? click to reveal ▸
POST = create a new resource (maps to "C" in CRUD). PUT = update an existing resource, usually identified by an id in the path (maps to "U"). POST goes to the collection (/posts); PUT targets one item (/posts/{id}).
Q3 · What does @PathVariable do? click to reveal ▸
It binds a segment of the URL path to a method parameter. In /posts/{id} the {id} placeholder is bound to the method's Long id argument — so a request to /posts/5 calls the method with id = 5.

The problem we're solving first

What's wrong with returning Entities directly?

In Session 2 our controller returned the same object that mapped to the database. That's convenient — but it leaks. When you return your Entity straight to the client, every field goes out, including ones the client should never see.

✗ Returning the Entity (User)

@GetMapping("/{id}")
public User getUser(@PathVariable Long id) {
    return userRepo.findById(id).get();
}

// The JSON that goes out:
{
  "id": 7,
  "name": "Ravi",
  "email": "ravi@tcs.com",
  "password": "$2a$10$x9...",  // leaked!
  "isAdmin": true,          // internal flag
  "failedLogins": 3       // internal
}

✓ Returning a DTO (UserDTO)

@GetMapping("/{id}")
public UserDTO getUser(@PathVariable Long id) {
    return userService.getUser(id);
}

// The JSON that goes out:
{
  "id": 7,
  "name": "Ravi",
  "email": "ravi@tcs.com",
  "role": "USER"
}

// Only what the client needs.
The core problem Your database table and your API response are two different things with two different audiences. Letting one dictate the other couples them together: change a column and you accidentally change (or break) your public API. The fix is a dedicated object for transfer — a DTO.

The solution

What is a DTO?

A DTO — Data Transfer Object — is a plain Java class whose only job is to carry data between layers, especially across the boundary between your application and the outside world. It has no JPA annotations and no business logic. It carries only what the client needs to send or receive. Three reasons make it worth the extra class:

🔒

Security

Hide sensitive fields. Passwords, internal flags and audit columns live on the Entity but never appear on the DTO — so they can't leak through the API.

🎯

Flexibility

Shape the response to the client's needs. One screen wants a summary, another wants detail — different DTOs, same Entity. Not every response needs every field.

🔗

Decoupling

Your DB schema can change without breaking the API contract. Rename a column or split a table — map it in the service, and clients never notice.

One-line definition to remember The Entity maps to the database. The DTO travels over the wire. They often have similar fields, but they exist for completely different reasons — and keeping them separate is what makes an API clean and safe to evolve.

The architectural anchor of this session

Layered architecture

A professional Spring Boot app is split into layers, each with one clear job. A request flows down through them and the response flows back up. This separation is what makes the code testable, maintainable, and easy for a team to work on in parallel.

Client browser · Postman · mobile app — sends an HTTP request
▼   HTTP request (JSON)
@RestController Controller receives the request, validates input, calls the Service with a DTO
▼   DTO
@Service Service business logic lives here · maps DTO ↔ Entity · decides what happens
▼   Entity
@Repository Repository talks to the database using the Entity · Spring Data writes the SQL
▼   SQL
Database stores and retrieves the actual data (H2 today)
Single Responsibility, layer by layer The Controller only deals with HTTP. The Service only deals with business rules. The Repository only deals with the database. No layer reaches past its neighbour — and that discipline is exactly what keeps a growing codebase from turning into spaghetti.

How DTO and Entity talk to each other

DTO mapping — the conversion

Because the Controller speaks DTO and the Repository speaks Entity, the Service sits in the middle and converts between them. We write two small helper methods for this — one each way.

// Entity  ->  DTO  (on the way OUT to the client)
private PostDTO mapToDTO(Post post) {
    PostDTO dto = new PostDTO();
    dto.setId(post.getId());
    dto.setTitle(post.getTitle());
    dto.setDescription(post.getDescription());
    dto.setContent(post.getContent());
    return dto;
}

// DTO  ->  Entity  (on the way IN from the client)
private Post mapToEntity(PostDTO dto) {
    Post post = new Post();
    post.setTitle(dto.getTitle());
    post.setDescription(dto.getDescription());
    post.setContent(dto.getContent());
    return post;
}

Doing this by hand vs automatically We map by hand today so the conversion is visible — you can see exactly which fields cross the boundary. In larger projects teams use libraries like MapStruct or ModelMapper to generate these mappers automatically, but the concept is identical.

Stop writing boilerplate

Lombok

Lombok is a library that generates repetitive Java code for you at compile time. Instead of hand-writing getters, setters, constructors and toString() on every class, you add one annotation and Lombok writes them. We'll use it on both our Entity and our DTO.

AnnotationWhat it generates
@DataGetters, setters, toString(), equals() and hashCode() — the all-in-one workhorse.
@AllArgsConstructorA constructor taking every field as an argument.
@NoArgsConstructorAn empty constructor — JPA and Jackson both need one.
@BuilderThe builder pattern: Post.builder().title("...").build().

✗ Without Lombok (~30 lines)

public class Post {
    private Long id;
    private String title;

    public Long getId() { return id; }
    public void setId(Long id) {
        this.id = id;
    }
    public String getTitle() {
        return title;
    }
    public void setTitle(String t) {
        this.title = t;
    }
    // ...and on, and on...
}

✓ With Lombok (5 lines)

@Data
public class Post {
    private Long id;
    private String title;
}

// @Data generates all the
// getters, setters, equals,
// hashCode and toString.
IDE setup gotcha Lombok works through annotation processing. If your IDE shows "cannot find method getTitle()", you simply need to enable annotation processing (IntelliJ) or install the Lombok plugin (Eclipse). The code is correct — the IDE just hasn't been told to run Lombok yet.

Why exception handling matters

What happens without custom exception handling?

In Session 2, asking for a missing id just returned null or a blank body. If your code throws instead, Spring's default is an ugly, leaky error. Here's what a client sees today when something goes wrong — and what we want them to see instead.

✗ The default (unhelpful)

⚠️ Response 500
{
  "timestamp": "2026-06-23T10:02...",
  "status": 500,
  "error": "Internal Server Error",
  "trace": "java.util.NoSuch...
    at PostService.get(Post...
    at ...300 more lines...",
  "path": "/api/posts/5"
}

Wrong status (it's not the server's fault), and a stack trace leaks internals.

✓ The target (clean)

✅ Response 404
{
  "timestamp": "2026-06-23T10:02...",
  "status": 404,
  "error": "Not Found",
  "message":
    "Post not found with id = 5",
  "path": "/api/posts/5"
}

Correct 404, a clear human message, no internals leaked.

The goal When a record isn't found, the client should get a 404 Not Found with a message like "Post not found with id = 5" — not a 500 with a wall of stack trace. We get there with a custom exception.

Building the clean 404

Custom exception — how it works

Three small pieces turn an ugly 500 into a clean 404. Each is just a few lines:

1

Extend RuntimeException

Create a class that extends RuntimeException and takes a message in its constructor, passing it to super(message).

2

Annotate @ResponseStatus

Put @ResponseStatus(HttpStatus.NOT_FOUND) on the class. Spring reads it and sets the HTTP response code to 404 automatically.

3

Throw it

In the service, throw it with orElseThrow(...) when a record is missing. Spring turns the exception into the response.

// Step 1 + Step 2: the exception class
@ResponseStatus(HttpStatus.NOT_FOUND)   // <-- tells Spring: respond 404
public class ResourceNotFoundException extends RuntimeException {

    public ResourceNotFoundException(String message) {
        super(message);                 // the message the client will see
    }
}

// Step 3: throwing it from the service
Post post = postRepository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException(
                "Post not found with id = " + id));

The clean way to handle "might not exist"

Optional and orElseThrow()

When you call findById(id), Spring Data returns an Optional<Post> — a container that may or may not hold a value. It exists to force you to handle the "not found" case, preventing the dreaded NullPointerException.

findById(id)
→
Optional<Post>
→
present → return the Post
Optional<Post>
→
empty → throw ResourceNotFoundException
Post post = postRepository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException(
                "Post not found with id = " + id));

// Reads almost like English:
// "find by id, or else throw this exception."
// If found -> 'post' holds it. If not -> exception is thrown.
Common mistake Don't call .get() directly on an Optional — if it's empty it throws a generic NoSuchElementException with no useful message. Always prefer orElseThrow() so you control the exception and the message the client receives.

A first taste — demo only

Spring Security — what it gives you

Just by adding one dependency to the classpath, Spring Security locks down your entire application with sensible defaults — before you write a single line of security code. Here's what happens automatically:

🚧

Blocks everything

Every endpoint now requires authentication. Hit any URL and you're stopped until you log in.

🔑

Login page

A default login form appears in the browser — no HTML written by you. Username defaults to user.

🎲

Generated password

A random password is printed in the console on startup. You use it to log in the first time.

You override the defaults with three lines in application.properties:

spring.security.user.name=admin
spring.security.user.password=admin123
# (the generated password is no longer used once these are set)
This slide is a demo walkthrough — no lab exercise We'll add the dependency and watch the login page appear live. That's it. Full security — JWT tokens, role-based access, custom config — is a dedicated later session. For now, just see how little it takes to switch protection on.

Setting up the bigger picture

Monolithic architecture — the problem

Everything we've built so far is a monolith: one application, one deployable unit, all modules bundled together. That's perfectly fine to start with — but as a system grows, the monolith starts to hurt.

One Big Application User Management · Orders · Payments · Inventory — all deployed together as a single unit
▼ deployed to
One Server / One Database
💥

One failure = total outage

If the Payments module crashes, it can take the whole application down with it. There's no isolation.

📈

Scale everything or nothing

Only Orders is under heavy load? Too bad — you must duplicate the entire app to scale that one part.

🐢

Risky deployments

A one-line change to Inventory means rebuilding and redeploying the entire application. Releases get slow and scary.

The alternative

Microservices architecture — the solution

Microservices break that one big app into many small, independent services, each owning one business capability and its own database. They communicate over the network (usually REST). Compare directly with the monolith:

User Service + its own DB
Order Service + its own DB
Payment Service + its own DB
Inventory Service + its own DB
🚀

Independent deployment

Update the Payment service and deploy just that one — the rest keep running untouched.

📊

Independent scaling

Order traffic spikes? Run more copies of only the Order service. Pay for what you actually need.

🛡️

Failure isolation

If Payments goes down, Users and Inventory keep serving. One failure no longer means total outage.

The trade-off (honest framing) Microservices aren't free — they add network calls, distributed data, and operational complexity. They shine for large systems and large teams. A small project is often better off as a well-structured monolith. Knowing when to use each is the real skill.

Two ways to handle more load

Horizontal vs vertical scaling

When traffic grows, you have two options to keep up. They sound similar but are fundamentally different in cost and ceiling.

⬆️ Vertical scaling (scale up)

1 server · 4 GB
▼ add resources
1 server · 64 GB + more CPU

Make a single machine more powerful. Simple, but there's a hard ceiling (you can't add RAM forever) and big machines get expensive fast. Still one point of failure.

➡️ Horizontal scaling (scale out)

server
server
server

Add more machines running the same service, behind a load balancer. No real ceiling, cost-effective with commodity hardware, and naturally fault-tolerant.

Why this matters for microservices Microservices are designed for horizontal scaling. Because each service is small and stateless, you can spin up as many copies as you need and spread traffic across them — exactly the model the cloud is built for.

Making scale-out actually work

Load balancing & high availability

If you run three copies of a service, something has to decide which copy each request goes to. That's a load balancer — it sits in front and distributes incoming traffic evenly across the instances.

⚖️

Load balancing

Spreads requests so no single instance is overwhelmed — and removes the single point of failure.

♻️

High availability

If Instance 2 dies, the load balancer simply stops sending it traffic. Instances 1 and 3 keep serving — users never notice.

Tools you'll meet later In the Spring world this is handled by Spring Cloud — with Eureka (a registry where services find each other) and an API Gateway (a smart front door that routes and load-balances). Just names for now; you'll use them in a later session.

Your closing cheatsheet

Session recap + annotation cheatsheet

Five concepts from today, in plain English:

  • DTO — a plain class that carries only what the client needs; keeps the Entity hidden.
  • Layered architecture — Controller → Service → Repository, each with one job.
  • Custom exception — your own RuntimeException with a meaningful message.
  • @ResponseStatus — maps that exception to the right HTTP code (e.g. 404) automatically.
  • orElseThrow() — clean way to throw when an Optional is empty; no null checks.
@EntityMarks a class as a database table (JPA).
@DataLombok — generates getters, setters, toString, equals, hashCode.
@RepositoryMarks the data-access layer; gives CRUD with no boilerplate.
@ServiceMarks the business-logic layer; managed by Spring.
@ResponseStatusSets the HTTP status code for a response or exception.
@AutowiredInjects a dependency (e.g. the repository into the service).
