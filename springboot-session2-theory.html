An annotation that marks a class as a web request handler whose return values are written directly into the HTTP response body (as JSON or text), rather than being treated as the name of a view/HTML page to render. It's what makes a class serve data to API clients.

The core idea of today

What is a web service?

A web service is a way for two applications to exchange data over a network, regardless of what language or platform either one is written in. The key word is data — a web service does not return a web page for a human to look at. It returns raw, structured data (usually JSON) for another program to read and use.

The one distinction that matters A traditional web app returns HTML — a page meant for a human's eyes in a browser. A web service returns data (JSON/XML) — meant for another application to consume. Same server technology underneath, completely different audience.

The language of modern APIs

JSON syntax

JSON (JavaScript Object Notation) is the format your API will send and receive. It's lightweight, human-readable, and every language can parse it. You'll be looking at exactly this in Postman in about 20 minutes — so let's make it land now. A single Student looks like this:

{
  "id": 1,
  "firstName": "Ravi",
  "lastName": "Sharma",
  "email": "ravi@tcs.com",
  "active": true
}
  • Curly braces { } wrap an object — one record, like one student.
  • Key–value pairs — the "key" (always a quoted string) is followed by a colon : then the value.
  • Commas separate the pairs — every pair except the last one ends with a comma.
  • Value types: strings in quotes, numbers bare, booleans as true/false.

Square brackets [ ] make an array — an ordered list. A response with several students is an array of objects:

[
  { "id": 1, "firstName": "Ravi",  "lastName": "Sharma" },
  { "id": 2, "firstName": "Anita", "lastName": "Verma" },
  { "id": 3, "firstName": "Karan", "lastName": "Patel" }
]
You won't write this by hand Spring Boot uses a library called Jackson to automatically convert your Java Student objects into JSON on the way out, and JSON back into Java objects on the way in. You return a Student; the client receives JSON. That conversion is called serialization.

Vocabulary you'll hear all the time

Key web service terminology

Four terms come up constantly when people talk about services. Each is simple — here's the one-line definition plus an everyday analogy so it sticks.

🍽️

Service Provider

The application that offers the service and answers requests. Analogy: the kitchen that cooks the food. Your Spring Boot API is the provider today.

📱

Service Consumer

The application that calls the service and uses its data. Analogy: the customer who orders. Postman, a browser, or a mobile app are consumers.

⏳

Synchronous

The consumer waits for the response before doing anything else. Analogy: standing at the counter until your order is handed over.

🔔

Asynchronous

The consumer doesn't wait — it carries on and is notified later. Analogy: taking a buzzer and sitting down until it goes off.

Loose coupling — the superpower Because the two sides only agree on the data format and the URLs, neither cares what language the other is written in. A Python app, a React frontend and an Android phone can all call the same Java API. Change one side's internals and the other doesn't break. That independence is called loose coupling, and it's why web services scale across teams and technologies.

Two styles of web service

REST vs SOAP

There are two main ways to build web services. SOAP is the older, heavier, XML-based standard. REST is the lighter, more flexible style that dominates modern APIs — and the one you'll use today. SOAP isn't the villain; it's just from an earlier era.

AspectSOAPREST
What it isA strict protocol with rigid rulesAn architectural style with guidelines
Data formatXML onlyJSON (usually), XML, others
Speed / weightHeavier, more verboseLightweight and fast
ComplexityHigher — lots of ceremonyLower — easy to learn
Contract (WSDL)Required — a formal WSDL fileNot required
TransportUses plain HTTP verbs over standard URLsUses HTTP methods (GET/POST/PUT/DELETE) fully
Where each lives today In industry, REST is the standard for modern APIs — web, mobile, microservices. SOAP still exists in legacy banking, payments and large enterprise/government systems where its strict contracts and built-in security were valued. You'll meet REST everywhere; you'll meet SOAP occasionally in older codebases.

What makes an API "RESTful"

The six REST principles

REST is defined by a set of constraints. You don't need to memorise all six today, but knowing they exist helps you understand why REST APIs are built the way they are.

  • Stateless — every request carries everything the server needs to handle it. The server keeps no memory of previous requests. This is the big one: it's why REST scales so well, because any server can handle any request.
  • Client–Server — the client (UI) and server (data) are separate and evolve independently. Neither needs to know the other's internals.
  • Uniform Interface — a consistent, predictable way to access resources: same URL patterns, same HTTP methods, every time.
  • Cacheable — responses can declare whether they may be stored and reused, reducing repeated work.
  • Layered System — there can be intermediaries (load balancers, gateways, caches) between client and server without either side knowing.
  • Code on Demand (optional) — the server may send executable code to the client. Rarely used; the only optional constraint.
If you remember one thing Stateless. Each request stands alone — the server doesn't remember who you are between calls. That's the principle that shapes how REST APIs are designed and why they handle huge traffic gracefully.

Verbs that map to actions

HTTP methods → CRUD

REST uses the standard HTTP methods as verbs that describe what you want to do to a resource. The four you'll use map directly onto the four database operations known as CRUD — Create, Read, Update, Delete.

HTTP methodCRUD operationMeaningStudent API example
GETReadFetch existing dataGET /students — list all students
POSTCreateAdd new dataPOST /students — add a new student
PUTUpdateModify existing dataPUT /students/{id} — update student 1
DELETEDeleteRemove dataDELETE /students/{id} — remove student 1
Real-world framing Think of a student record: you GET it to view, POST a new one when a student enrols, PUT to fix their email, and DELETE when they leave. Same noun (/students), different verb for each action. That's the whole mental model of a REST API.

How the server reports the outcome

HTTP status codes

Every HTTP response carries a numeric status code telling the client what happened. There are dozens, but you only need these five for today's work. They're grouped: 2xx = success, 4xx = the client made a mistake, 5xx = the server broke.

200
OK
Success. The request worked. Spring Boot returns this by default for a successful GET that returns data.
201
Created
Resource created. Used after a successful POST. You'll set it yourself with @ResponseStatus(HttpStatus.CREATED) in the lab.
400
Bad Request
Client error. The request was malformed — e.g. invalid JSON in the body. Spring returns this when it can't parse what you sent.
404
Not Found
Not found. The URL or resource doesn't exist — e.g. asking for a student ID that isn't there.
500
Server Error
Server crashed. Something threw an unhandled exception inside your code. We'll learn to handle these gracefully in Session 3.

What makes a good endpoint

URI design principles

A URI (the path part of a URL) names a resource. The golden rule of REST: URIs are nouns, HTTP methods are verbs. Don't put the action in the URL — let the HTTP method (GET/POST/PUT/DELETE) say what you're doing. The URL just names what you're acting on.

✓ Good (nouns)

  • /students — the collection
  • /students/{id} — one student by ID
  • /students?name=John — filter with a query parameter
vs

✗ Bad (verbs in URL)

  • /getStudent — verb belongs in the method
  • /deleteStudentById — redundant & clumsy
  • /student/fetch — action in the path
The pattern to internalise Use plural nouns for collections (/students), a path segment to identify one item (/students/5), and query parameters for filtering or optional criteria (?name=John). The verb is never in the URL — it's the HTTP method.

A closer look

@RestController unpacked

You used @RestController in Session 1, but here's what it actually is under the hood. It's a convenience annotation that bundles two others together:

@RestController
= @Controller  +  @ResponseBody

@ResponseBody is the important half: it tells Spring to take whatever your method returns and write it directly into the HTTP response body as JSON (via Jackson), instead of treating the return value as the name of an HTML page to render. With a plain @Controller you'd have to add @ResponseBody to every single method:

✗ Plain @Controller

@Controller
public class StudentController {

    @GetMapping("/students")
    @ResponseBody  // needed on
    public List<Student> all() {
        return students;
    }
    // ...and on every
    // other method too
}

✓ @RestController

@RestController
public class StudentController {

    @GetMapping("/students")
    // no @ResponseBody needed
    public List<Student> all() {
        return students;
    }
    // applies to ALL methods
    // automatically
}

Keeping URLs DRY

@RequestMapping at class level

Putting @RequestMapping("/students") on the class sets a common URL prefix for every method inside. Each method's mapping is then relative to that prefix. This keeps your code clean and your URLs consistent — you write /students once instead of repeating it everywhere.

Without class-level mapping

@GetMapping("/students")
@GetMapping("/students/all")
@GetMapping("/students/{id}")
@PostMapping("/students")
@PutMapping("/students/{id}")
→

With @RequestMapping("/students")

// class: @RequestMapping("/students")
@GetMapping
@GetMapping("/all")
@GetMapping("/{id}")
@PostMapping
@PutMapping("/{id}")
How the URL is assembled Spring joins the class prefix and the method path: class /students + method /{id} = the full endpoint /students/{id}. Change the prefix in one place and every endpoint moves with it.

How you'll call your API

Testing tools: Postman + Swagger

You can't easily test a POST or PUT from a browser address bar — browsers only do GET. So we use proper API testing tools. You'll use both today; they complement each other.

🚀

A dedicated desktop app for calling APIs. Full control: pick the HTTP method, set headers, paste a JSON body. This is the industry-standard tool teams use daily. Best for testing POST/PUT/DELETE with custom bodies.

Taking control of the response

ResponseEntity

By default, when you return a Student from a method, Spring sends it back with status 200 OK. But sometimes you need control over the full HTTP response — the status code, the headers, and the body. That's what ResponseEntity<T> gives you: it's a wrapper around the entire response.

Raw return (always 200)

@GetMapping("/{id}")
public Student getById(
        @PathVariable int id) {
    return findStudent(id);
    // status is always 200,
    // even if not found
}

ResponseEntity (you choose)

@GetMapping("/{id}")
public ResponseEntity<Student> getById(
        @PathVariable int id) {
    Student s = findStudent(id);
    if (s != null)
        return ResponseEntity.ok(s);   // 200
    return ResponseEntity
            .notFound().build();        // 404
}
When to reach for it Use ResponseEntity when you need to return different status codes depending on the outcome — 201 Created after a POST, 404 Not Found when something's missing, or to attach custom headers. It's the professional way to build precise REST responses.

Calling another service (trainer demo)

RestTemplate

So far your app is the provider — it answers requests. But sometimes a Spring Boot app needs to be a consumer too, calling another REST API from inside its own code. RestTemplate is Spring's classic synchronous client for exactly that.

RestTemplate restTemplate = new RestTemplate();

// Call another service and map its JSON straight into a Student
Student student = restTemplate.getForObject(
        "http://localhost:8080/students/1",
        Student.class);

System.out.println(student.getFirstName());
  • getForObject(url, Type.class) sends a GET request and converts the JSON response directly into a Java object.
  • It's synchronous — your code waits for the response before continuing.
  • There are matching methods: postForObject, put, delete, and more.
Trainer demo only — don't code this today We'll see this in action as a quick demonstration. It becomes far more relevant in a microservices context, where many small services call each other — a topic for a later session. For now, just understand what it's for.
