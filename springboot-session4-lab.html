⚠️ Spring Cloud ↔ Spring Boot version compatibility — read this first

This is the #1 cause of failure in this session. Spring Cloud releases are tied to specific Spring Boot versions. On start.spring.io the versions are matched for you automatically — but if you set them manually, use this guide:

Spring BootSpring Cloud release train
3.3.x2023.0.x (Leyton)
3.2.x2023.0.x (Leyton)
3.1.x2022.0.x (Kilburn)
2.7.x2021.0.x (Jubilee)

Symptom of a mismatch: the app fails to start, or @EnableEurekaServer/the gateway classes won't resolve. Fix: align the versions per this table, or just regenerate from Spring Initializr which always picks a compatible pair.

The labs — build the microservices system
LAB 1:8761▶ Create the Eureka Server
Concept The Eureka Server is the service registry — the phonebook every other service registers with. It's almost pure config: one annotation and four properties.
  1. Spring Initializr (start.spring.io): Maven · Java 17 · Group com.tcs.training · Artifact eureka-server.
  2. Dependency: Eureka Server (spring-cloud-starter-netflix-eureka-server). That's the only one.
  3. Add @EnableEurekaServer to the main class:
package com.tcs.training.eurekaserver;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.netflix.eureka.server.EnableEurekaServer;

@SpringBootApplication
@EnableEurekaServer                  // turns this app INTO the registry
public class EurekaServerApplication {

    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
  1. Configure src/main/resources/application.properties:
server.port=8761
spring.application.name=eureka-server
eureka.client.register-with-eureka=false
eureka.client.fetch-registry=false
server.port=8761The standard Eureka port — clients are configured to expect it here.
spring.application.name=eureka-serverA friendly name for this app.
eureka.client.register-with-eureka=falseThe server shouldn't register itself in its own registry.
eureka.client.fetch-registry=falseIt doesn't pull a registry from elsewhere — it IS the registry.
  1. Run the app and open http://localhost:8761 in the browser.

LAB 2:8080GET▶ Create the Student Microservice
Concept A normal Spring Boot REST app that registers itself with Eureka. The only new ingredient is the Eureka Client dependency plus the defaultZone property pointing at the registry.
  1. Spring Initializr: Artifact student-ms. Dependencies: Spring Web + Eureka Discovery Client + Spring Boot DevTools.
  2. Create StudentController.java. It injects Environment so the response shows which instance answered — essential for seeing load balancing later.
package com.tcs.training.studentms.controller;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.core.env.Environment;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/students")
public class StudentController {

    @Autowired
    private Environment environment;     // lets us read the actual running port

    @GetMapping
    public String getStudents() {
        return "Student Service - Port: "
                + environment.getProperty("local.server.port");
    }
}
  1. Configure application.properties:
server.port=8080
spring.application.name=STUDENT-MS
eureka.client.service-url.defaultZone=http://localhost:8761/eureka/
Critical — the service name spring.application.name=STUDENT-MS must be exactly this (uppercase by convention). The Gateway will route to this precise name with lb://STUDENT-MS. A typo here means the gateway route silently fails to resolve.
  1. Run the app, then refresh localhost:8761.

LAB 4:8082 / :8083▶ Run Multiple Instances of Student MS
Concept Right now Student MS runs on 8080. We'll start two more instances of the same project on 8082 and 8083 to simulate horizontal scaling — without creating new projects. We override the port at runtime with a VM argument.
IntelliJ IDEA
  1. Run → Edit Configurations…
  2. Select the Student MS config → Copy it (the duplicate icon)
  3. Rename to "Student MS · 8082"
  4. In VM options add: -Dserver.port=8082
    (if VM options is hidden: Modify options → Add VM options)
  5. Apply → Run. Repeat for -Dserver.port=8083
Eclipse / STS
  1. Right-click project → Run As → Run Configurations…
  2. New Maven Build
  3. Goals: spring-boot:run
  4. VM Arguments: -Dserver.port=8082
  5. Apply → Run. Repeat for 8083
Command-line alternative (any IDE) From the student-ms folder you can also run: 
mvn spring-boot:run -Dspring-boot.run.jvmArguments="-Dserver.port=8082"
Open a second terminal and repeat with 8083.

LAB 5:9393▶ Create the API Gateway
Concept The Gateway is the single entry point. It has no controller — it's configuration only. It registers with Eureka (so it can discover services) and we'll add routing in the next lab.
  1. Spring Initializr: Artifact api-gateway. Dependencies: Gateway (spring-cloud-starter-gateway) + Eureka Discovery Client.
  2. Configure the base application.properties:
server.port=9393
spring.application.name=api-gateway
eureka.client.service-url.defaultZone=http://localhost:8761/eureka/
  1. Run the Gateway and refresh Eureka — API-GATEWAY should now be registered too.
Expected The gateway starts on 9393 and appears in the Eureka dashboard alongside STUDENT-MS and CUSTOMER-MS. There's nothing to call yet — routes come next.
SymptomCauseFix
Startup warning / failure about web application typeSpring Cloud Gateway is built on the reactive (WebFlux) stack — a Web MVC dependency can clash.Add spring.main.web-application-type=reactive to application.properties, and make sure spring-boot-starter-web is NOT also included.
Why no spring-boot-starter-web? Spring Cloud Gateway uses the reactive Netty server, not Tomcat. If you accidentally add the regular Web starter too, they conflict. The Gateway starter brings everything the gateway needs.
LAB 6▶ Manual Route Configuration
Concept — what a route is A route maps an incoming URL pattern on the gateway to a backend service. It has three parts: an id (any name), a uri (which service to forward to), and one or more predicates (which incoming URLs trigger it).
  1. Add the routing config to the Gateway's application.properties:
spring.cloud.gateway.routes[0].id=student-ms-route
spring.cloud.gateway.routes[0].uri=lb://STUDENT-MS
spring.cloud.gateway.routes[0].predicates[0]=Path=/students-api/**

spring.cloud.gateway.routes[1].id=customer-ms-route
spring.cloud.gateway.routes[1].uri=lb://CUSTOMER-MS
spring.cloud.gateway.routes[1].predicates[0]=Path=/customers-api/**
What lb:// means lb = Load Balancer. uri=lb://STUDENT-MS tells the gateway: don't use a hardcoded address — ask Eureka for the live instances of STUDENT-MS and load-balance across them. This is what makes round robin work in the next lab.
What the predicate means Path=/students-api/** means: "any request to the gateway whose path starts with /students-api/ gets forwarded to this route's service." The part after is passed along — so :9393/students-api/students → STUDENT-MS + /students.
  1. Restart the Gateway so it picks up the new routes.
Path mapping in plain terms Request GET :9393/students-api/students → predicate matches /students-api/** → gateway strips nothing by default and forwards the full path to STUDENT-MS, which serves /students. So your controller's @RequestMapping("/students") lines up with the part after /students-api.

  2. Hit it 3–4 times rapidly. Watch the port in the response string change each time.
Expected — Round Robin in action The response cycles through the instances:
Request 1 → Student Service - Port: 8080 Request 2 → Student Service - Port: 8082 Request 3 → Student Service - Port: 8083 Request 4 → Student Service - Port: 8080 (loops back)
Watch the consoles too Tile the IDE consoles for all three Student MS instances. As you hit the endpoint, watch which console logs the incoming request — it rotates 8080 → 8082 → 8083. This is the visual payoff of the whole session: one URL, traffic spread automatically across three instances by the gateway.
  1. Test Customer MS too: GET http://localhost:9393/customers-api/customers
Expected — Customer MS via gateway
Student Service - Port: 8080 / 8082 / 8083 (rotating) Customer Service - Port: 8081 (only one instance)
Both services are now reachable through the single gateway port 9393.

LAB 8▶ Discovery Locator Approach
Concept Instead of defining each route by hand, Discovery Locator tells the gateway to auto-create routes for every service registered in Eureka, keyed by service name. Zero route config — but the service name appears in the URL.
  1. Comment out the manual routes from Lab 6 (prefix each line with #).
  2. Add the discovery locator config:
spring.cloud.gateway.discovery.locator.enabled=true
spring.cloud.gateway.discovery.locator.lower-case-service-id=true
  1. Restart the Gateway.
  2. Test with the service name in the path:
    • GET http://localhost:9393/student-ms/students
    • GET http://localhost:9393/customer-ms/customers
Expected Both work with no manual route configuration. Student MS still round-robins across its three instances — load balancing is automatic here too. Note the URL now uses /student-ms/ (lower-cased service id) instead of /students-api/.
Manual routes vs Discovery Locator — the trade-off Manual routes → clean, custom URLs (/students-api/...), full control, more config; better for production. Discovery Locator → automatic, zero config, but service names leak into URLs (/student-ms/...); better for development & exploration. The trade-off is URL cleanliness vs zero configuration.
BONUS▶ Observe the Eureka Heartbeat — do this if you finish early
Concept Eureka relies on heartbeats. If a service stops sending them, Eureka eventually removes it — and the gateway stops routing to it. Let's watch that self-healing happen.
  1. Stop one Student MS instance — e.g. the one on port 8082.
  2. Wait 30–60 seconds (Eureka has a grace period before it evicts a missing instance).
  3. Refresh the Eureka dashboard. STUDENT-MS drops from 3 instances to 2.
  4. Hit the gateway endpoint repeatedly again.
Expected — self-healing routing
Before: 8080 → 8082 → 8083 → 8080 ... After : 8080 → 8083 → 8080 → 8083 ... (8082 gone, no errors)
Traffic now only cycles between 8080 and 8083. No config change was needed — Eureka detected the missing heartbeat and the gateway adjusted automatically.
Why this matters In production, instances crash, get redeployed, or scale down all the time. This heartbeat-driven eviction is what keeps traffic flowing to healthy instances only — the foundation of high availability, with zero manual intervention.
