Session 3 recap

Let's warm up the brain. Read each question, decide your answer in your head, then click the card to reveal it. If all three feel obvious, you're ready for today.

Q1 · What is a DTO and why use it? click to reveal ▸
A Data Transfer Object — a plain Java class that carries only the data the client needs. We use it to hide sensitive fields (security), shape responses (flexibility), and decouple the API contract from the database schema. The Entity maps to the DB; the DTO travels over the wire.
Q2 · What does @ResponseStatus do on a custom exception? click to reveal ▸
It tells Spring which HTTP status code to return when that exception is thrown. Putting @ResponseStatus(HttpStatus.NOT_FOUND) on ResourceNotFoundException makes Spring respond with a clean 404 automatically — no try/catch needed.
Q3 · What does Spring Security do the moment its dependency is added? click to reveal ▸
It locks down every endpoint with sensible defaults: a login page appears, a random password is printed in the console, and the default username is user — all before you write a single line of security code.

A 60-second anchor

Monolith vs Microservices — revisited

Last session we discussed why microservices exist. Today we build the infrastructure that runs them. Quick reminder of the contrast before we dive in:

🧱 Monolith

  • One deployable unit — everything ships together.
  • One shared database.
  • One failure = full outage — no isolation.
  • Scale it = duplicate the whole app.

🧩 Microservices

  • Independent services, deployed separately.
  • A database per service.
  • Failure isolation — one down, others up.
  • Scale only the service that needs it.
Today's framing Microservices give you independence and scalability — but they introduce a new question: how do all these separate, independently-deployed services find and talk to each other? That's the problem the rest of this session solves.

Why the next patterns exist

Horizontal vs vertical scaling — revisited

⬆️ Vertical (scale up)

One machine, more RAM/CPU. Simple, but has a hard ceiling and gets expensive.

➡️ Horizontal (scale out)

More machines running the same service. No real ceiling, cost-effective, fault-tolerant.

Here's the catch — and it sets up the whole session Horizontal scaling means you'll have MULTIPLE INSTANCES of the same service (e.g. Student MS on ports 8080, 8082, 8083). That creates two brand-new problems:
  • How does the client know all the addresses? Instances come and go on dynamic ports.
  • How does traffic get distributed evenly across them?
These two problems are solved by the next two patterns: a Service Registry and an API Gateway.

The vocabulary of microservices

The 5 microservices design patterns

Five patterns come up in every real microservices system. Each solves one specific pain. We'll build the first two today (Registry + Gateway) and cover the other three conceptually — you'll meet all of them in industry projects.

⚙️

Config Server

✗ 10 services, 10 properties files ✓ one central Git-backed config
📇

Service Registry

✗ dynamic ports, can't hardcode ✓ a live catalog of what's running
🚪

API Gateway

✗ clients shouldn't know internal URLs ✓ one single entry point
🔍

Distributed Tracing

✗ request spans 5 services, what's slow? ✓ end-to-end request tracking
🔌

Circuit Breaker

✗ Service B down, A keeps calling ✓ detect, stop, return fallback
Today's scope Build: Service Registry (Eureka) + API Gateway (Spring Cloud Gateway). Concept-only: Config Server, Distributed Tracing, Circuit Breaker. Knowing they exist — and what problem each solves — is what matters today.

Pattern 1 of 5 — concept only

Config Server

Imagine 10 microservices, each with its own application.properties. You need to change the database URL — now you're editing 10 files and redeploying 10 services. The Config Server fixes this: every service pulls its configuration from one central, Git-backed source at startup.

Git Repository all config files, version-controlled
▼ served by
Spring Cloud Config Server
▼ services read from it at startup
Service A
Service B
Service C
When you'll use it When you have environment-specific configs (dev / test / prod) across many services, and want a single, audited place to manage them. Change one property in Git, and services can refresh without a redeploy. Not building this today — but you'll encounter it in real projects.

Pattern 2 of 5 — building this today

Service Registry (Eureka) — how it works

A Service Registry is a live phonebook of every running service: its name, address, and port. Netflix Eureka is the registry we'll use. Services register themselves on startup and prove they're alive with a heartbeat. Here's the flow:

1

Eureka starts

The registry boots on port 8761 — the phonebook itself, empty at first.

2

Student MS registers

On startup it tells Eureka "I'm STUDENT-MS at port 8080" and sends a heartbeat every ~30s.

3

Customer MS registers

Same thing — CUSTOMER-MS joins the registry. Now Eureka knows both.

4

Gateway asks Eureka

"Where is STUDENT-MS?" → Eureka returns the live list of instances.

The heartbeat — self-healing built in Registered services ping Eureka regularly. If a service crashes and the heartbeats stop, Eureka removes it from the registry after a grace period — so the gateway stops routing traffic to a dead instance, automatically. No manual cleanup, no config change.

Pattern 3 of 5 — building this today

API Gateway — what it does

The API Gateway is the single front door to your whole system. Clients talk to one URL; the gateway figures out the rest. It has three core jobs:

🧭

Routing

Forward /students-api/** to Student MS and /customers-api/** to Customer MS, based on the URL.

⚖️

Load Balancing

Spread traffic across multiple instances of a service (8080, 8082, 8083) so none is overwhelmed.

🚪

Single Entry Point

Clients only ever know one URL — the gateway (port 9393). Internal services stay hidden.

A gateway route in application.properties looks like this:

spring.cloud.gateway.routes[0].id=student-ms-route
spring.cloud.gateway.routes[0].uri=lb://STUDENT-MS          # lb = load-balanced via Eureka
spring.cloud.gateway.routes[0].predicates[0]=Path=/students-api/**

Secondary capabilities (mention-only today): authentication, rate limiting, request logging — all natural to centralise at the gateway.

How traffic gets spread

Load balancing — Round Robin

When several instances of a service are running, the gateway needs a rule to pick one per request. The simplest and default rule is Round Robin: just cycle through the instances in order, one after another, then loop back to the start.

Request 1 →
:8080 ✓
:8082
:8083
Request 2 →
:8080
:8082 ✓
:8083
Request 3 →
:8080
:8082
:8083 ✓
Request 4 →
:8080 ✓
:8082
:8083
The magic word: lb:// The lb:// prefix in your gateway route (uri=lb://STUDENT-MS) is what switches this on. lb = load balancer. It tells Spring Cloud: don't use a hardcoded address — ask Eureka for all live instances of STUDENT-MS and round-robin across them automatically. You'll watch this happen live in the lab.

Pattern 4 of 5 — concept only

Distributed Tracing

When one request hops through five services and the whole thing is slow, which service caused it? Distributed tracing answers that. Every request gets a unique trace ID that's passed along the chain; each service logs with that ID, and a tool like Zipkin stitches them into one timeline.

Tooling & scope In the Spring world this is done with Micrometer Tracing + Zipkin. Use case: "the request takes 3 seconds — who's responsible?" Conceptual overview only today, but it's invaluable the moment you have more than two services.

Pattern 5 of 5 — concept only

Circuit Breaker

If Service B is down, Service A shouldn't keep hammering it — that wastes resources and can cascade into a system-wide failure. A circuit breaker wraps the call and watches for failures, just like an electrical breaker that trips to prevent damage. It has three states:

CLOSED

Normal operation. Requests flow through to Service B. Failures are counted.

OPEN

Too many failures — the breaker trips. Requests are rejected instantly and a fallback is returned. No more hammering.

HALF-OPEN

After a timeout, let one trial request through. If it succeeds → back to CLOSED. If it fails → back to OPEN.

CLOSED
— failures →
OPEN
— timeout →
HALF-OPEN
— success →
CLOSED
Analogy & tooling Like a household circuit breaker: it trips to prevent damage, then lets you check if it's safe to resume. In Spring, this is implemented with Resilience4j. The fallback might return cached data or a friendly "try again later" instead of an error cascade. Not building today.

Config that matters — the registry

Eureka Server setup — key config

The Eureka Server is almost all configuration. One annotation on the main class, four lines of properties:

@SpringBootApplication
@EnableEurekaServer                 // <-- turns this app INTO the registry
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
server.port=8761                              # the standard Eureka port — clients expect it
spring.application.name=eureka-server
eureka.client.register-with-eureka=false      # the server shouldn't register with itself
eureka.client.fetch-registry=false            # it doesn't fetch a registry — it IS the registry
Why those two false lines? Eureka components are clients by default. For the server we switch both off: it has no reason to register itself in its own phonebook, and no reason to download a registry from elsewhere. Leave these out and you'll see noisy startup errors as it tries to talk to a registry that doesn't exist.

Config that matters — a microservice

Eureka Client setup — key config

Any microservice that wants to join the registry just needs the Eureka Client dependency and three properties. No annotation is required in modern Spring Cloud — it auto-detects the client from the classpath (older versions used @EnableEurekaClient).

server.port=8080                                                  # override at runtime for more instances
spring.application.name=STUDENT-MS                                # the service ID — Gateway routes to THIS name
eureka.client.service-url.defaultZone=http://localhost:8761/eureka/   # where to register
The single most important line spring.application.name=STUDENT-MS is critical. This exact name is what Eureka lists the service under, and what the Gateway uses to route (lb://STUDENT-MS). Get it wrong or mismatched and the gateway route silently fails to resolve. Convention here: uppercase service IDs.
Running more instances server.port=8080 is just a default. At runtime you override it with a VM argument — -Dserver.port=8082 — to start the same service on another port. Each one registers under STUDENT-MS, and Eureka groups them as multiple instances of one service.

Two ways to configure the gateway

API Gateway — manual routes vs discovery locator

There are two ways to tell the gateway how to route. You'll try both in the lab.

Manual routing

spring.cloud.gateway.routes[0]
  .id=student-ms-route
spring.cloud.gateway.routes[0]
  .uri=lb://STUDENT-MS
spring.cloud.gateway.routes[0]
  .predicates[0]=Path=/students-api/**
  • Clean, custom URLs (/students-api/...)
  • More control (filters, rewrites)
  • More config to write per service

Discovery locator

spring.cloud.gateway.discovery
  .locator.enabled=true
spring.cloud.gateway.discovery
  .locator.lower-case-service-id=true
  • One block, auto-routes all Eureka services
  • Zero per-service config
  • Service name appears in URL (/student-ms/...)
Which to use Manual routes for production-grade, clean public APIs where you control the URL shape. Discovery locator for quick prototyping and exploration — zero config, but it exposes internal service names in the URL. The trade-off is URL cleanliness vs zero configuration.

Simulating horizontal scaling locally

Running multiple instances

To see load balancing, you need more than one instance of Student MS. You don't create new projects — you run the same project again on a different port using a VM argument.

IntelliJ IDEA

  1. Run → Edit Configurations…
  2. Copy the Student MS config, rename it "Student MS · 8082"
  3. Add VM option: -Dserver.port=8082
  4. Repeat for 8083, then run both

Eclipse / STS

  1. Right-click project → Run As → Maven build…
  2. Goals: spring-boot:run
  3. VM args: -Dserver.port=8082
  4. Apply → Run; repeat for 8083
