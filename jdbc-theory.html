The motivating problem

The persistence problem

Picture your app: an ArrayList<Student> with 1000 students added through the day. Then the server restarts — a deploy, a crash, a power blip. Where's your data now?

While the JVM runs

List<Student> list = new ArrayList<>();
list.add(new Student(1, "Ravi"));
list.add(new Student(2, "Anita"));
list.add(new Student(3, "Karan"));
list.add(new Student(4, "Priya"));
list.add(new Student(5, "Arjun"));

System.out.println(list.size());
// 5  — all present

After the server restarts

List<Student> list = new ArrayList<>();
// the old objects lived in RAM.
// RAM is gone. They're gone too.




System.out.println(list.size());
// 0  — everything lost
The one idea this whole session rests on In-memory data lives only as long as the JVM lives. A database persists data beyond the process lifecycle — it survives restarts, crashes, and redeploys. JDBC is the bridge from your Java objects to that durable store.

The core idea

What is JDBC?

JDBC (Java Database Connectivity) is Java's standard API for talking to relational databases. The crucial thing to understand: it's mostly a set of interfaces, and the database vendor supplies the actual code.

🧩

Just interfaces

A set of Java interfaces in the java.sql package — not a framework. They define what operations exist, not how they work.

📦

Vendor provides the driver

The database vendor ships a JAR file — the driver — that implements those interfaces for their database. One JAR per database.

🏗️

The foundation of ORMs

Every modern DB framework — JPA, Hibernate, Spring Data — sits on top of JDBC. They generate JDBC calls for you.

Why interface-based design matters here Because your code talks to interfaces, it stays the same whether the database underneath is MySQL, Oracle, PostgreSQL, or Derby. Swap the driver JAR and the connection URL — your logic is untouched. Learn JDBC today, and Spring Data JPA will never feel like magic again.

How the pieces fit together

JDBC architecture

A request flows down through these layers to the database, and the results flow back up. Each layer has exactly one job:

Java Application your code — calls the JDBC API
↕
JDBC API the java.sql interfaces — Connection, PreparedStatement, ResultSet
↕
DriverManager finds and loads the right driver for your URL
↕
JDBC Driver the vendor JAR — the concrete implementation
↕
Database MySQL · Derby · H2 · PostgreSQL · Oracle
The key handoff Your code only ever touches the top two layers — your classes and the java.sql interfaces. DriverManager is the matchmaker: you hand it a URL like jdbc:derby:..., and it picks the driver that knows how to speak that database's protocol.

A bit of history you should recognise

The four driver types

JDBC drivers come in four historical "types." You only ever use Type 4 in modern work — but interviewers ask, and legacy code exists, so know the landscape:

TypeNameHow it worksStatus
1JDBC-ODBC BridgeConverts JDBC calls into ODBC calls via a native bridgeRemoved (Java 8)
2Native APIUses the DB's native C/C++ client libraries — needs a local installRare
3Network ProtocolRoutes calls through a middleware server that talks to the DBLegacy enterprise
4Thin / Pure JavaPure-Java JAR that speaks the DB's wire protocol directlyIndustry standard
The only one that matters today Type 4 is what you add to pom.xml (or the build path). It's a single JAR, needs no local database client installed, and works everywhere the JVM runs. MySQL Connector/J, the Derby driver, the H2 driver — all Type 4. This is all you'll use in modern development.

Your toolkit for the lab

The four core interfaces

Almost every JDBC program is built from just these four. You'll see all of them in code within the next 15 minutes — here's the one-line version of each:

🗂️

DriverManager

A utility class. You give it a URL + credentials; it finds the matching driver and hands back a Connection.

🔌

Connection

One open session with the database. Creating it opens a real network socket — so it's precious. Always close it.

📝

PreparedStatement

A pre-compiled, parameterized SQL query with ? placeholders. Safe and fast. Use it for everything.

📊

ResultSet

A cursor over the rows a SELECT returns. rs.next() walks forward one row at a time until there are none left.

A mental picture DriverManager gets you a Connection. From the connection you create a PreparedStatement. Running a SELECT on it gives you a ResultSet to read row by row. That chain is the spine of every JDBC program.

The pattern behind every JDBC program — keep this in mind all lab

The five JDBC steps

1
Register the driver — automatic in JDBC 4.0+, shown for understanding
Class.forName("org.apache.derby.jdbc.EmbeddedDriver");
2
Open a connection
Connection con = DriverManager.getConnection(url, user, pass);
3
Prepare the statement
PreparedStatement ps = con.prepareStatement(sql);
4
Execute
ps.executeUpdate();   // for INSERT/UPDATE/DELETE
ps.executeQuery();    // for SELECT -> returns a ResultSet
5
Close / release resources
con.close();          // or use try-with-resources (slide 14)
This slide stays relevant for the entire hands-on section Every JDBC program you write — INSERT, SELECT, UPDATE, DELETE — follows these same five steps. Memorise the shape and the lab becomes pattern-matching.

Two ways to send SQL — only one is acceptable

Statement vs PreparedStatement

✗ Statement — string concatenation

String sql = "INSERT INTO employees "
  + "VALUES (" + id + ", '"
  + first + "', '" + last + "')";

Statement st = con.createStatement();
st.executeUpdate(sql);
// recompiled every call,
// messy, and INJECTABLE

✓ PreparedStatement — placeholders

String sql = "INSERT INTO employees "
  + "VALUES (?, ?, ?)";

PreparedStatement ps =
    con.prepareStatement(sql);
ps.setInt(1, id);
ps.setString(2, first);
ps.setString(3, last);
ps.executeUpdate();
// compiled once, bound safely
⚡

Performance

The SQL template is pre-compiled once by the DB, then reused with different parameters.

👀

Readability

No quote-juggling or string soup. The query shape is clear; values are bound separately.

🛡️

Security

Immune to SQL injection — parameters are never treated as SQL. (Next two slides.)

The most memorable five minutes of this session

SQL injection — the attack

Imagine a login query built the lazy way, with string concatenation:

String sql = "SELECT * FROM users WHERE username = '" + username
           + "' AND password = '" + password + "'";

With normal input it works fine. But watch what a malicious user types into the username field:

username = admin' --

// The query string becomes:
SELECT * FROM users WHERE username = 'admin' --' AND password = 'anything'

The -- starts a SQL comment, so the entire password check is commented out. The attacker is logged in as admin with no password. And it gets worse:

username = '; DROP TABLE users; --     // goodbye, entire table
Why you must care SQL injection is consistently in the OWASP Top 10 most critical web vulnerabilities, decade after decade. It's how real companies get breached. And one line of PreparedStatement code prevents all of it.

The fix is almost embarrassingly simple

SQL injection — the defense

✗ Vulnerable — Statement

String sql =
  "SELECT * FROM users WHERE "
  + "username = '" + username
  + "' AND password = '"
  + password + "'";

Statement st = con.createStatement();
ResultSet rs = st.executeQuery(sql);
// input becomes SQL syntax

✓ Safe — PreparedStatement

String sql =
  "SELECT * FROM users WHERE "
  + "username = ? AND password = ?";

PreparedStatement ps =
    con.prepareStatement(sql);
ps.setString(1, username);
ps.setString(2, password);
ResultSet rs = ps.executeQuery();
// input is always just a value
How setString() defeats the attack The driver sends the SQL template and the values separately. Parameters are never parsed as SQL — they're bound as literal data. So the attacker's admin' -- is treated as a username that literally contains those characters. The database searches for a user named admin' --, finds none, and login fails. The quote can't "escape" out of the value.
The rule — never forget it Never use Statement for any query that includes user input. Always use PreparedStatement. No exceptions.

A small distinction that trips up everyone

executeQuery vs executeUpdate

Two execute methods, two purposes. Pick the wrong one and you'll either get no data or an exception — a classic beginner bug.

MethodUsed forReturns
executeQuery()SELECT — reading dataa ResultSet (the rows)
executeUpdate()INSERT, UPDATE, DELETE, CREATE — changing data/schemaan int — the number of rows affected
The common mistake Using executeUpdate() for a SELECT returns 0, not your data. Using executeQuery() for an INSERT throws an exception. Getting this pairing wrong is one of the most frequent JDBC bugs for beginners — match the method to the SQL verb.
Bonus: the return value of executeUpdate() is useful The row count it returns tells you whether anything actually changed. UPDATE ... WHERE emp_id = 999 on a missing id returns 0 — that's how you distinguish "ran fine but matched nothing" from "found it and changed it."

Catch the right thing

Exception handling — SQLException

JDBC methods throw SQLException (a checked exception). Don't swallow it with a generic catch(Exception e) — SQLException carries three pieces of diagnostic gold that a plain Exception doesn't:

try {
    // ... JDBC code ...
} catch (SQLException e) {
    System.out.println("Message  : " + e.getMessage());      // human-readable
    System.out.println("SQLState : " + e.getSQLState());     // standard 5-char code
    System.out.println("ErrorCode: " + e.getErrorCode());    // vendor-specific int
}
💬

getMessage()

A human-readable description of what went wrong.

🔢

getSQLState()

A standardized 5-character code, portable across databases.

🏷️

getErrorCode()

A vendor-specific integer for the exact error.

A few SQLState codes you'll actually meet 42X05 — table not found  ·  23505 — duplicate primary key  ·  08001 — cannot connect to the database. These are exactly what your DevOps team needs when diagnosing a production DB issue.

The modern, leak-proof pattern

try-with-resources

A Connection is a real network socket. If an exception fires before you close it, the connection leaks — and enough leaks exhaust the database. The old finally dance is verbose and easy to botch. try-with-resources auto-closes for you.

✗ Old — manual finally

Connection con = null;
try {
    con = DriverManager.getConnection(...);
    // ... work ...
} catch (SQLException e) {
    e.printStackTrace();
} finally {
    if (con != null) {
        try { con.close(); }
        catch (SQLException ex) {}
    }
}
// 15+ lines, easy to forget

✓ Modern — try-with-resources

String sql = "INSERT INTO ... VALUES (?,?)";

try (Connection con =
         DriverManager.getConnection(...);
     PreparedStatement ps =
         con.prepareStatement(sql)) {

    // ... work ...

} catch (SQLException e) {
    e.printStackTrace();
}
// both auto-closed, always
The rule Connection, Statement/PreparedStatement and ResultSet all implement AutoCloseable. Declare them in the try(...) header and they close automatically — on success and on exception, in reverse order. Always use try-with-resources in production code.

Organising the code properly

The DAO pattern

Mixing connection code, SQL, and Scanner menus in one class becomes a tangle fast. The DAO (Data Access Object) pattern splits responsibilities into three clean layers:

🔌

DBConnector

One class, one method: getConnection(). No SQL, no UI — just hands back a connection.

🗄️

DAO

All SQL operations as methods: insert(), getAll(), getById(), update(), delete(). No UI code.

🖥️

Tester / Main

The Scanner menu, calls DAO methods, displays results. No SQL here.

Tester / Main UI & input
→
EmployeeDAO all SQL
→
DBConnector the connection
→
Database
Why it pays off — and the JPA connection Switching from Derby to MySQL? You change only the DBConnector URL and driver (and maybe minor SQL syntax). Business logic and UI are untouched. Spring Data's JpaRepository IS a DAO — when you call repository.findById(), it runs the exact JDBC you're about to write, generated for you.

Where this fits in the bigger picture

JDBC vs Spring Data JPA

You're not learning JDBC to use it forever — you're learning it so the abstractions above it make sense. Here's the honest comparison:

AspectRaw JDBCSpring Data JPA
Code volumeHigh — manual SQL + ResultSet mappingLow — method names or annotations
ControlFull — write any SQL you wantModerate — JPQL + @Query for custom
Learning curveLow — just SQL + 4 interfacesHigher — JPA concepts needed first
Best forLegacy systems, complex hand-tuned SQLMost modern applications
The takeaway You learn JDBC to understand the foundation. You use Spring Data JPA to build efficiently on top of it. Every repository.save() you'll ever write is JDBC running underneath.
