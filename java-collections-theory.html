The motivating problem

Why collections exist

Arrays are fine until they aren't. Three everyday scenarios show where plain arrays fall short:

📈

Store 500 names…

String[] s = new String[500] — and now you need a 501st. You're stuck; arrays are fixed-size.

✂️

Remove item at index 200

Arrays: manually shift every later element left by one. Collections: one method call — list.remove(200).

🔎

Is "Ravi" already in it?

Arrays: write a loop. Collections: list.contains("Ravi").

The bottom line Arrays have a fixed size and no built-in utilities. Collections are resizable, feature-rich, and type-safe via generics. They exist to solve exactly these problems.

The reference map for the whole section

The Collections hierarchy

java.lang.Iterable
   └── java.util.Collection
         ├── List      → ArrayList, LinkedList, Vector
         ├── Set       → HashSet, LinkedHashSet, TreeSet
         └── Queue     → LinkedList, ArrayDeque

java.util.Map  (separate hierarchy — NOT a Collection)
         → HashMap, LinkedHashMap, TreeMap
List — ordered, indexed, duplicates Set — no duplicates Queue — FIFO/LIFO ends Map — key → value
The one thing to remember Map is NOT a subtype of Collection. It stores key–value pairs rather than individual elements, which is why it sits in its own separate hierarchy.

The type-safety layer under every collection

Generics

✗ Without generics (raw type)

ArrayList list = new ArrayList();
list.add("Ravi");
list.add(101);   // no error — anything goes

String n = (String) list.get(1);
// ClassCastException at RUNTIME!

✓ With generics (type-safe)

ArrayList<String> list = new ArrayList<>();
list.add("Ravi");
list.add(101);   // COMPILE error — caught now

String n = list.get(0);
// no cast needed
🛡️

Compile-time checks

Type errors are caught before the program ever runs.

🎯

No casting

The compiler already knows the element type.

📖

Self-documenting

<String> states intent right in the declaration.

Rule Always specify the type parameter. The diamond <> lets the compiler infer it from the left side. Raw types exist only for backward compatibility with pre-Java-5 code.

Lists — ordered, indexed, allow duplicates

ArrayList — the default List

A List preserves insertion order, supports index access, and allows duplicates. ArrayList is its workhorse — backed by a resizable array (initial capacity 10; when full it grows to ~1.5× and copies elements over).

Everyday operations

ArrayList<String> d = new ArrayList<>();
d.add("Eng");
d.add(1, "Mktg");  // insert at index
d.get(0);          // "Eng"
d.set(0, "HR");    // replace
d.contains("HR");   // true
d.remove("Mktg");    // by value
d.remove(0);         // by index

Performance

get(i)O(1) — fast
add(end)O(1) amortized
add/remove(mid)O(n) — shifts
contains()O(n) — scan

Use it when you read often, modify rarely — the go-to default.

Classic gotcha For ArrayList<Integer>, remove(1) removes by index, not value. To remove the value 1, use remove(Integer.valueOf(1)).

The doubly-linked alternative

LinkedList — nodes & a Deque

A LinkedList is a chain of nodes, each holding the data plus prev and next references. Inserting in the middle just re-points two links — no shifting.

null ← [prev|"Ravi"|next] ⇄ [prev|"Priya"|next] ⇄ [prev|"Arjun"|next] → null
        HEAD                                              TAIL

It's also a Deque (queue + stack)

// Queue — FIFO
q.offer("T1");    // add to tail
q.poll();         // remove head
q.peek();         // read head

// Stack — LIFO
st.push("F1");   // add to head
st.pop();         // remove head

Performance

get(i)O(n) — traverse
add/remove(head/tail)O(1)
memory/elementhigher (2 refs)
Modern advice For pure queue/stack use, prefer ArrayDeque — better cache performance. Reach for LinkedList only when you genuinely need List + Queue behaviour together.

The decision guide

ArrayList vs LinkedList

OperationArrayListLinkedListWinner
Random access by indexO(1)O(n)ArrayList
Insert / delete at beginningO(n)O(1)LinkedList
Insert at endO(1) amort.O(1)Tie
Memory per elementLowHigherArrayList
Cache performanceExcellentPoorArrayList
In practice Use ArrayList by default — it wins in most real-world scenarios. Use LinkedList only when you constantly add/remove at the ends or need a queue/deque. ~90% of the time, ArrayList is the right answer.

Two ways to define order

Sorting — Comparable vs Comparator

Comparable — natural order, inside the class

class Employee implements
        Comparable<Employee> {
  @Override
  public int compareTo(Employee o) {
    // neg / 0 / pos
    return Double.compare(salary, o.salary);
  }
}
Collections.sort(employees); // by salary

Comparator — flexible, outside the class

// lambda — by name
employees.sort((a, b) ->
    a.name.compareTo(b.name));

// by salary, descending
employees.sort(Comparator
    .comparingDouble(Employee::getSalary)
    .reversed());

// dept, then salary
employees.sort(Comparator
    .comparing(Employee::getDept)
    .thenComparingDouble(Employee::getSalary));
Which to use Comparable when there's ONE obvious natural ordering (and you own the class). Comparator when you need multiple orderings, or to sort a class you can't modify (like String or Integer).

Three ways to create a list

resizechange
Arrays.asList()NoYes

List.of() = truly immutable (Java 9+). Arrays.asList() = fixed size but elements editable. Need both? Wrap: new ArrayList<>(List.of(...)).

Don't confuse them Collections (plural, the utility class) vs Collection (singular, the interface) — two completely different things.

Sets — no duplicates, no index

Set & how HashSet works

A Set rejects duplicates and has no positional access — perfect for the "have I seen this already?" problem, answered in O(1) instead of a List's O(n) scan.

Membership in O(1)

Set<String> seen = new HashSet<>();
seen.add("Mumbai");
seen.add("Mumbai");  // ignored
seen.contains("Mumbai"); // instant
seen.size();             // 1

The hashing mechanism

On add("Ravi"): compute "Ravi".hashCode() → map to a bucket index → store there. On contains(): same hash → check only that one bucket. That's why lookups are instant — and why iteration order is unpredictable (elements sit in hash-order buckets, not insertion order).

The hashCode/equals contract For custom objects in a HashSet (or as HashMap keys), you must override both hashCode() and equals() consistently: equal objects must return the same hashCode. Override one, always override the other.

Three flavours of Set

Choosing a Set implementation

⚡

HashSet

Fastest, no order. O(1) add/remove/contains. The default.

🔗

LinkedHashSet

Keeps insertion order via a linked list. Slightly more memory.

🌳

TreeSet

Always sorted (Red-Black tree). O(log n). Adds first(), last(), floor(), ceiling().

Set<Integer> ts = new TreeSet<>(List.of(85, 72, 91, 68, 72));
System.out.println(ts);   // [68, 72, 85, 91] — sorted, duplicate dropped
OperationHashSetLinkedHashSetTreeSet
add / remove / containsO(1) avgO(1) avgO(log n)
Iteration orderNoneInsertionSorted

The most useful collection in real projects

Map & HashMap

A Map stores unique keys each mapped to one value — a dictionary, a phone book, a student-ID-to-grade record. HashMap uses the same hashing trick as HashSet for O(1) lookups.

HashMap<String, Integer> pop = new HashMap<>();
pop.put("Mumbai", 20_000_000);
pop.put("Mumbai", 21_000_000);   // updates existing key
pop.get("Mumbai");                  // 21000000
pop.get("Chennai");                 // null — absent
pop.getOrDefault("Chennai", 0);   // 0
pop.containsKey("Delhi");
Frequency counter — a pattern you'll use everywhere 
for (String w : words)
    freq.put(w, freq.getOrDefault(w, 0) + 1);
// {apple=3, banana=2, cherry=1}

Reading a Map — and its ordered cousins

Iterating Maps

entrySet() — both key & value (best)

for (Map.Entry<String,Integer> e
        : pop.entrySet()) {
    e.getKey();
    e.getValue();
}

// cleanest (Java 8+):
pop.forEach((city, p) ->
    System.out.println(city + " → " + p));

Also: keySet() / values()

for (String k : pop.keySet()) { }
for (Integer v : pop.values()) { }
Don't Loop keySet() then call get(key) inside — that's two lookups per row. Use entrySet().
ImplementationOrder of keys
HashMapNone (fastest)
LinkedHashMapInsertion order
TreeMapSorted by key (O(log n))

Section 2 · How objects relate — IS-A

Inheritance (IS-A)

A subclass IS-A kind of its superclass. Dog extends Animal means a Dog is an Animal and inherits its behaviour. This is what enables polymorphism.

Inherit & override

class Animal {
  void eat() { ... }
}
class Dog extends Animal {
  @Override
  void eat() { ... }  // new behaviour
  void bark() { ... }
}

Polymorphism & super

Animal a = new Dog(); // Dog IS-A Animal
a.eat();   // runs Dog's version

// in Manager extends Employee:
super(name, salary);  // parent ctor
super.calculatePay(); // parent method
Two rules worth keeping Always use @Override — without it, a signature typo silently creates a new method instead of overriding. And use inheritance only for a genuine, stable IS-A relationship — not merely to reuse code (that's what composition is for).

HAS-A — the relationship freshers most often confuse

Composition vs Aggregation

Composition — strong ownership

class House {
  private Room living;
  House() {
    living = new Room(); // House CREATES it
  }
}

House owns the Room. Destroy the House → the Room goes too. A Room without a House makes no sense.

Aggregation — weak ownership

class Department {
  private List<Employee> emps;
  void add(Employee e) {
    emps.add(e); // e existed before
  }
}

Department references Employees. Disband it → Employees still exist and can move elsewhere.

The decision question Can the contained object exist and make sense without the container? Yes → Aggregation. No → Composition.

A core design principle

Favour composition over inheritance

You want a Logger that writes to a file and a database. Java has no multiple inheritance — you can't extends FileWriter, DatabaseConnector. Composition solves it cleanly:

class Logger {
  private FileWriter file;          // HAS-A
  private DatabaseConnector db;     // HAS-A

  void log(String msg) {
    file.write(msg);
    db.insert("logs", msg);
  }
}

No limits

Compose as many collaborators as you need.

Swappable

Change behaviour at runtime — inject a different writer.

Focused

Classes stay small and single-purpose.

Testable

Inject mock dependencies easily.

The rule Inherit for a genuine IS-A relationship. Compose to reuse behaviour. Most of the time, composition is the right answer.

Two ways to define a contract

Abstract class vs Interface

Abstract class — partial implementation

abstract class Vehicle {
  void start() { ... }       // shared
  abstract double mileage(); // must impl
}
class ElectricCar extends Vehicle {
  @Override double mileage() { ... }
}

Interface — a capability contract

interface Saveable {
  void save(String f);
  default void backup() { ... } // Java 8+
}
class Report implements
        Printable, Saveable { ... } // multiple!
Abstract classInterface
Constructor / instance fieldsYesNo (only constants)
MethodsAbstract + concreteAbstract + default + static
Multiple inheritanceNo (one extends)Yes (many implements)
Which to use Abstract class when classes share state and behaviour. Interface to define a capability unrelated classes might share — like Comparable, Runnable, Serializable.

Section 3 · Data beyond the JVM

Why File I/O

Collections keep data alive in memory; databases keep it on a server. Files keep it on disk — simple, no server needed, perfect for configs, logs, and exports.

Byte streams

InputStream / OutputStream — for binary data: images, audio, PDFs.

Character streams

Reader / Writer — for text, with encoding handled for you.

Rule of thumb You'll work with text ~90% of the time — so reach for character streams. They handle character encoding (UTF-8, etc.) automatically.
The File class — paths only, no content java.io.File represents a path. It checks exists(), isFile(), length(), and creates/deletes files & directories — but it does not read or write content. Use File.separator (or Paths.get(...)) instead of hardcoding \ or /.

Reading & writing text files

Buffered text I/O

Wrap a FileReader/FileWriter in a buffered layer — it reads/writes large chunks at once instead of one slow character at a time. PrintWriter adds println()/printf() convenience on top.

Write — BufferedWriter / PrintWriter

try (PrintWriter pw = new PrintWriter(
       new BufferedWriter(
         new FileWriter("emp.txt")))) {
  pw.println("ID,Name,Salary");
  pw.printf("%d,%s,%.2f%n",
            101, "Ravi", 75000.0);
}  // auto-flush + close

Read — BufferedReader, line by line

try (BufferedReader br = new BufferedReader(
       new FileReader("emp.txt"))) {
  String line;
  while ((line = br.readLine()) != null) {
    String[] p = line.split(",");
    // p[0]=ID, p[1]=Name ...
  }
}  // readLine() returns null at EOF
The wrapping idea BufferedReader wraps FileReader wraps the file. Each layer adds a capability — FileReader opens it, BufferedReader adds buffering + readLine(). Scanner is an easier alternative for token-based reading (nextInt(), nextLine()), but slower for large files.

Safety, errors & the modern API

try-with-resources, NIO.2 & Properties

Always auto-close + handle IOException

File f = new File("config.txt");
if (!f.exists()) { /* defaults */ }

try (BufferedReader br =
       new BufferedReader(new FileReader(f))) {
  // ...
} catch (IOException e) {
  System.out.println(e.getMessage());
}

All streams implement AutoCloseable — forgetting to close leaks the file handle and can lock it.

NIO.2 (Java 7+) — concise

Path p = Paths.get("out.txt");
Files.writeString(p, "Hello\n");
String all = Files.readString(p);
List<String> lines =
    Files.readAllLines(p);

For new code prefer java.nio.file; know java.io for legacy code.

Properties files — exactly how Spring Boot reads config 
Properties props = new Properties();
try (FileInputStream fis = new FileInputStream("config.properties")) {
    props.load(fis);
}
String url = props.getProperty("db.url");   // like application.properties!

The synthesis — it all works together

Collections + File I/O in one flow

Read a CSV into a List, process it with Collections, write the result back — the practical payoff of everything in this session.

// 1. READ a CSV into a List of objects
List<Employee> emps = new ArrayList<>();
try (BufferedReader br = new BufferedReader(new FileReader("employees.csv"))) {
    br.readLine();                       // skip header
    String line;
    while ((line = br.readLine()) != null) {
        String[] p = line.split(",");
        emps.add(new Employee(Integer.parseInt(p[0].trim()), p[1].trim(),
                              p[2].trim(), Double.parseDouble(p[3].trim())));
    }
}
// 2. PROCESS with Collections
emps.sort(Comparator.comparingDouble(Employee::getSalary).reversed());
Map<String,Long> byDept = new HashMap<>();
for (Employee e : emps) byDept.merge(e.getDept(), 1L, Long::sum);

// 3. WRITE the result back out
try (PrintWriter pw = new PrintWriter(new BufferedWriter(new FileWriter("report.csv")))) {
    pw.println("Name,Department,Salary");
    for (Employee e : emps)
        pw.printf("%s,%s,%.2f%n", e.getName(), e.getDept(), e.getSalary());
}
Everything in one workflow ArrayList, sort, Comparator, HashMap, BufferedReader, BufferedWriter and PrintWriter — all cooperating to read, transform, and persist data.

Everything in one view

Session recap

Lists

  • ArrayList — default; O(1) index, O(n) middle insert
  • LinkedList — O(1) ends; also a Deque
  • Comparable = natural order; Comparator = flexible

Sets

  • HashSet fastest, no order
  • LinkedHashSet insertion order
  • TreeSet sorted, O(log n)

Maps

  • HashMap fast/unordered, TreeMap sorted
  • Iterate with entrySet()
  • Override hashCode()+equals() for custom keys

Class relationships

  • IS-A = inheritance; HAS-A = composition/aggregation
  • Favour composition for reuse
  • Abstract = partial impl; Interface = contract

File I/O

  • BufferedReader/Writer for text; PrintWriter for formatting
  • NIO.2 Files for concise modern code
  • Properties = how Spring reads config

Golden rules

  • Always use generics <Type>
  • ArrayList & HashMap are the right default ~90% of the time
  • Always use try-with-resources for any stream
