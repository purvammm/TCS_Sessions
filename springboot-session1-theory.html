The "before" picture

Life before Spring Boot

The Spring Framework arrived in 2003 and changed Java backend development for the better — it introduced dependency injection and a clean way to structure applications. But while Spring made well-organised code possible, starting and assembling a Spring project remained slow and error-prone. Before you could write a single line of business logic, you had to do a great deal of manual plumbing:

  • Write reams of XML to declare every object (bean) and describe how they connect to one another.
  • Manually wire dependencies — explicitly tell Spring which object should be handed to which other object.
  • Install and configure an external Tomcat server, then build a WAR file and deploy it into that server.
  • Hand-pick compatible library versions — Spring MVC, Jackson, the logging library, the servlet API — and hope they all agreed with each other.
  • Copy the same boilerplate setup into every new project, every single time.
The real cost Teams routinely spent a day or more just getting an empty project to start up correctly. The framework was powerful, but the setup ceremony stood between developers and the actual work of building features. Spring Boot was created in 2014 to eliminate exactly this friction.

Pain point #1 — configuration

XML configuration hell

In classic Spring, the container had no way to discover your objects automatically — you described every one of them in an XML file. Even a tiny application's configuration looked like this:

<beans xmlns="http://www.springframework.org/schema/beans">
    <bean id="greetingService" class="com.example.GreetingServiceImpl" />

    <bean id="helloController" class="com.example.HelloController">
        <property name="greetingService" ref="greetingService" />
    </bean>

    <bean class="org.springframework.web.servlet.view.InternalResourceViewResolver">
        <property name="prefix" value="/WEB-INF/views/" />
        <property name="suffix" value=".jsp" />
    </bean>
</beans>
  • Every new class meant another <bean> entry, added and maintained by hand.
  • A typo in a class name or a ref wasn't caught by the compiler — it failed only at runtime, often deep into startup.
  • On real projects the XML grew into hundreds of lines, frequently larger and harder to maintain than the Java it was wiring together.
Why this matters Spring Boot replaces nearly all of this with annotations and auto-configuration. The XML above becomes a couple of @Service / @RestController annotations the framework discovers automatically — which is exactly what you'll write in the lab.

Pain points #2, #3 & #4

Wiring, servers & version juggling

🔌

Manual wiring

You connected every dependency yourself in XML. Forget one and you discovered it only when the app crashed at runtime, not at compile time.

🖥️

No embedded server

Build a WAR → install Tomcat → configure it → deploy the WAR into it. Four slow, manual steps before the app would even respond.

🧩

Version conflicts

You aligned versions of Spring, Jackson, logging and the servlet API by hand. Mismatches caused cryptic errors — the infamous "dependency hell."

📝

Repeated boilerplate

The same scaffolding was copied into every project before any feature work could begin — wasted, repetitive effort.

The pattern behind all four Every one of these is configuration work that is the same for almost every project. Spring Boot's insight was simple but powerful: if 95% of projects configure these things the same way, the framework should do it for you by default and let you override only the 5% that's different. That idea is called convention over configuration, and it's the heart of Spring Boot.

The solution

What is Spring Boot?

Spring Boot is a layer built on top of the Spring Framework that makes it easy to create stand-alone, production-grade applications you can simply run. The crucial point: it does not replace Spring. You still write Spring beans, controllers and services exactly as before — Boot simply removes the manual setup that used to surround them.

Your Application controllers, services, business logic
▼ runs on
Spring Boot auto-config · embedded server · starters · Actuator
▼ built on
Spring Framework IoC container · MVC · data access · AOP
Mental model Spring is the engine; Spring Boot is the ignition and auto-tuning that gets the engine running instantly. Everything you learn about core Spring — beans, dependency injection, MVC — still applies. Boot just means you spend your time on features instead of setup.

What Spring Boot gives you

The four things Boot provides

Everything Spring Boot does can be grouped into four capabilities. Understanding these four is understanding Spring Boot itself — we'll examine each in detail over the next slides.

⚙️

1 · Auto-configuration

Inspects your classpath and configures sensible beans automatically — replacing the old XML entirely.

🚀

2 · Embedded server

Tomcat is built into the application. Run it as an ordinary Java program; no external server to install.

📦

3 · Starter dependencies

One curated dependency brings in everything needed for a capability, with versions guaranteed to fit together.

📊

4 · Production-ready

Health checks, metrics and monitoring are available out of the box through Actuator.

Together these turn "a day of setup" into "run the main method."

The guiding principle

Convention over configuration

This is the philosophy that makes everything else possible. Rather than forcing you to describe every detail, Spring Boot assumes the most common, sensible setup (the "convention") and lets you override only the parts that are genuinely different for your project (the "configuration"). The framework makes intelligent guesses based on what it can see:

  • Sees spring-boot-starter-web on the classpath → assumes you're building a web app and starts an embedded Tomcat listening on port 8080.
  • Sees an H2 database driver → assumes an in-memory database and auto-configures a DataSource connection for you.
  • Finds a class annotated @RestController → registers it automatically to handle incoming web requests.
What this means for you day to day You write configuration only when your needs differ from the convention — for example, adding server.port=9090 to application.properties to change the port. If you're happy with the defaults (and usually you are), you write nothing at all. Less code to write, less code to get wrong.

How auto-configuration works

Auto-configuration, step by step

Auto-configuration is the mechanism that replaced XML. At startup, Spring Boot runs a decision process across hundreds of pre-written configuration classes, each one asking "should I activate, given this particular project?"

Scan classpath which jars are present?
→
Check conditions @ConditionalOnClass / OnMissingBean
→
Configure beans only what's needed
  • Each auto-configuration is conditional: it activates only if the relevant library is actually on the classpath (@ConditionalOnClass) and you haven't already defined that bean yourself (@ConditionalOnMissingBean).
  • This is why simply adding a starter is enough to get working features — the matching auto-configuration detects it and wires everything up.
  • If you define your own bean of the same type, Boot detects it and backs off, using yours instead. You're never locked in.
Debugging tip for later Run the app with --debug and Spring Boot prints an "auto-configuration report" listing every configuration it considered, which ones matched, and why. This is invaluable when something isn't being configured the way you expect.

Run it like any Java program

The embedded server

Traditionally a Java web app was a WAR file deployed into a separately-installed server. Spring Boot flips this around: the web server is packaged inside your application, so there is nothing external to install, configure, or deploy into.

  • Running the main method starts your application and its own web server in one step.
  • Everything packages into a single executable "fat" JAR — your code, all its dependencies, and the server together in one file.
  • You deploy by copying that one file to any machine with Java and running java -jar app.jar.
  • Tomcat is the default; switching to Jetty or Undertow is a one-dependency change.

✗ Traditional

  • Build a WAR
  • Install Tomcat
  • Configure it
  • Deploy into it
→

✓ Spring Boot

  • Build one JAR
  • Server is inside it
  • java -jar app.jar
  • Done

One line, many dependencies

Starter dependencies

A starter is a curated, ready-made bundle of dependencies for one capability. Instead of researching and version-matching a dozen separate libraries, you declare a single starter and trust Spring Boot to bring in everything that belongs together:

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
starter-web 1 line
→
Spring MVC
Jackson (JSON)
Validation
Embedded Tomcat
Why this ends "dependency hell" The versions inside a starter are chosen and tested by the Spring team to be mutually compatible. You don't specify versions at all — they're inherited from the Spring Boot parent. Upgrade Boot's version once and every managed library moves together, safely.

There are dozens of official starters, all following the naming pattern spring-boot-starter-*. These are the ones you'll encounter most often:

The fourth pillar

Production-ready features

Building an app is only half the job — in production you must be able to observe and manage it: is it healthy, how much memory is it using, how many requests is it handling? Spring Boot's Actuator module adds these management endpoints with almost no effort on your part.

  • /actuator/health — reports whether the app and its dependencies (database, etc.) are healthy. Load balancers poll this to decide whether to send traffic.
  • /actuator/metrics — memory, CPU, garbage collection, request counts and timings.
  • /actuator/info — build version and custom application details.
  • These feed monitoring dashboards and alerting tools (Prometheus, Grafana) that ops teams rely on.
Today's takeaway We won't configure Actuator today — it's covered in Session 3. The important point for now is conceptual: production concerns are built into Spring Boot, not bolted on later. This is what "production-grade" in the definition really means.

Request flow: Browser → HTTP Request → DispatcherServlet → Controller → Service → Repository → Database.
The key idea — read this carefully One central DispatcherServlet (Spring's "front controller") receives every incoming request and routes it to the correct controller method based on the URL. The controller calls a service for the business logic; the service calls a repository for data; the repository talks to the database. The response then travels back along the same path. Each box has exactly one job — this is called separation of concerns, and it's what keeps applications testable and easy to change.

Separation of concerns

Responsibilities by layer

Spring applications are organised into layers, each with a single, well-defined responsibility. This structure isn't enforced by the framework — it's a convention that keeps every part small, focused, and independently testable. A request passes down through the layers and the response comes back up.

🌐 ControllerReceives the HTTP request, reads input (path, query, body), calls a service, returns a response. Knows about the web — but contains no business rules.
🧠 ServiceThe brains of the app: business logic, calculations, validation, decisions. Knows nothing about HTTP or how data is stored.
🗃️ RepositoryTalks to the database — save, find, update, delete. Pure data access, no business logic.
💽 DatabaseStores the data persistently so it survives restarts.
Why split it up? Because each layer talks only to the one directly below it, you can change one without breaking the others. Swap the database, redesign the web API, or unit-test the service in isolation — the boundaries protect you. Today's lab builds the Controller and Service layers; the Repository and Database arrive in Session 2.

The entry point

@SpringBootApplication

Every Spring Boot application has exactly one main class carrying this annotation. Running its ordinary Java main method is what boots the entire application into life:

@SpringBootApplication
public class SpringbootDemoApplication {

    public static void main(String[] args) {
        SpringApplication.run(SpringbootDemoApplication.class, args);
    }
}

The single line SpringApplication.run(...) does a remarkable amount of work behind the scenes, in order:

  • Creates the IoC container (the application context).
  • Runs auto-configuration based on the classpath.
  • Scans your packages for components and registers them as beans.
  • Starts the embedded Tomcat server.

By the time main returns, your API is live and listening on port 8080.

Three annotations in one

@SpringBootApplication, unpacked

It looks like magic, but @SpringBootApplication is simply a convenience annotation that bundles three others together. Knowing what each does removes the mystery:

@Configuration
Marks the class as a source of bean definitions — Spring can read configuration and define beans from it.
@EnableAutoConfiguration
Switches on Spring Boot's auto-configuration mechanism, based on what libraries are on the classpath.
@ComponentScan
Scans this class's package and all sub-packages for components (controllers, services, repositories) and registers them as beans.
Why package placement matters (remember this!) Because @ComponentScan starts scanning from the main class's package downward, your controllers and services must live in that package or a sub-package of it. Put a controller in an unrelated package and Spring will never find it — your endpoint silently returns 404 with no error. This is one of the most common beginner bugs.

A core Spring idea

Dependency Injection: the why

Almost every object needs other objects to do its job — a controller needs a service, a service needs a repository. The question is: who creates those collaborators? Dependency Injection (DI) says an object should not create its own collaborators. Instead, they are created elsewhere and "injected" (handed in) from outside.

  • Looser coupling — a class depends on what it needs (an interface or type), not on how that thing is constructed. It doesn't care which concrete implementation it receives.
  • Easier testing — in a unit test you can inject a fake or mock dependency instead of the real one, testing your class in isolation.
  • Less plumbing — Spring's IoC container creates the objects (beans) and supplies them automatically, so you write far less wiring code.
  • Single responsibility — a class focuses on its own job rather than also managing the lifecycle of its dependencies.
One-line definition "Don't call us, we'll call you." Your class simply declares what it needs; the framework is responsible for providing it. This reversal of who's in control is exactly what "Inversion of Control" means.

The difference DI makes

Tight vs loose coupling

✗ Tight coupling (new)

public class HelloController {
    private GreetingService svc =
        new GreetingService();
    // hard-wired to ONE class:
    // can't swap or mock it
}

✓ Loose coupling (injected)

public class HelloController {
    @Autowired
    private GreetingService svc;
    // Spring supplies it:
    // swappable & testable
}

When the controller writes new GreetingService(), it is permanently glued to that one exact class — you can never substitute a different implementation, a mock for testing, or an enhanced version, without editing the controller's source code. When the service is injected instead, the controller simply asks for "a GreetingService" and Spring decides what to provide. You can swap implementations, wrap them, or mock them freely — the controller never changes.

Real-world payoff Imagine you later need to log every greeting, or fetch greetings from a database. With loose coupling you just provide a different bean — no controller changes. With new, you'd be editing every class that created the service. Loose coupling is what makes large codebases maintainable.

The vocabulary

Components, stereotypes & @Autowired

For Spring to inject a bean, it must first know about that bean. You tell it by annotating your classes. There are two halves to the story — registering a bean, and requesting one:

  • @Component — "Spring, please create and manage an instance of this class." During component scanning it becomes a bean held in the container.
  • @Autowired — "Spring, inject a matching managed bean right here." Spring finds a bean of the required type and supplies it.

The stereotype annotations below are all specialised forms of @Component. They behave the same way for registration, but each signals the class's role, which makes code self-documenting (and some add extra behaviour):

AnnotationLayerUse it for
@RestControllerWebHandling HTTP requests and returning data directly
@ServiceBusinessBusiness logic & rules
@RepositoryDataDatabase access (also translates DB exceptions)
@ComponentGenericAnything else Spring should manage

Who does the wiring?

The IoC container

Inversion of Control (IoC) is the principle; the IoC container (also called the application context) is the part of Spring that implements it. Instead of your code controlling when and how objects are created, you hand that responsibility to the container. At startup it scans for your annotated components, creates one instance of each (a singleton by default), works out what each one depends on, and injects those dependencies automatically.

@RestController
HelloController
@Service
GreetingService
@Repository
DataRepository
↑ Spring creates each bean once and injects it wherever @Autowired asks for that type

So in our app, the container creates the GreetingService bean once, sees that HelloController needs a GreetingService, and hands the same instance over. You never wrote new — that's "inversion": the framework calls and assembles your code, rather than your code assembling the framework.

Three ways to inject

Field, constructor & setter injection

✓ Constructor (recommended)

private final GreetingService svc;

public HelloController(
        GreetingService svc) {
    this.svc = svc;
}

Field (simplest)

@Autowired
private GreetingService svc;

Spring can inject a dependency in three places, and it's worth knowing the trade-offs:

  • Constructor injection — the dependency is passed into the constructor and stored in a final field. It can never be null and never change after construction, the class is easy to instantiate in tests, and required dependencies are obvious. This is the modern best practice. (With one constructor, @Autowired is even optional.)
  • Field injection — the shortest to write; Spring sets the field directly. We use it in today's lab purely for clarity, but it's harder to test and hides required dependencies.
  • Setter injection — Spring calls a setter method. Best for optional dependencies that can be changed after the object is built.

The style behind our endpoints

What is REST?

REST (REpresentational State Transfer) is the dominant architectural style for web APIs. The core idea is to model your application as a set of resources — nouns — each addressable by a URL, and to act on them using the standard HTTP methods — verbs.

  • A resource is a thing: a user, an order, a product, a greeting. It lives at a URL such as /users/42.
  • You act on a resource with an HTTP verb — GET to read, POST to create, PUT to update, DELETE to remove.
  • Each request is stateless — it carries everything the server needs, so the server keeps no memory of previous requests. This is what makes REST APIs easy to scale.
  • Responses are usually JSON; today we return plain text to keep the first example simple.

Verbs and their annotations

HTTP methods & mappings

Every interaction with a REST API uses an HTTP method that expresses intent. Spring gives each verb a dedicated annotation that maps a URL to one of your Java methods. Choosing the right verb is part of designing a clean API — for instance, a GET must never modify data, so it's safe to retry or cache.

HTTP methodPurposeSpring annotation
GETRead data — safe, makes no changes@GetMapping
POSTCreate new data@PostMapping
PUTUpdate / replace existing data@PutMapping
DELETERemove data@DeleteMapping
Today vs Session 2 Today's lab uses only @GetMapping because reading data is the simplest case — you test it just by visiting a URL in your browser. The other three verbs (which change data) come together with the database in Session 2, when we build full CRUD: Create, Read, Update, Delete.

Tour of the generated project

Project structure

When you unzip the generated project you'll find this layout. It follows the standard Maven convention — once you know it, every Spring Boot project looks familiar:

springboot-demo/ ├── src/main/java/ # your application code │ └── com/tcs/training/ │ └── SpringbootDemoApplication.java # entry point (main) ├── src/main/resources/ # config & static files │ └── application.properties # runtime config ├── src/test/java/ # tests ├── pom.xml # dependencies & build ├── mvnw / mvnw.cmd # Maven wrapper — run Maven w/o installing └── .gitignore
  • src/main/java holds your code; src/main/resources holds configuration and static assets; src/test/java holds tests — a clean separation.
  • The main class sits at the top of your package so component scanning reaches everything beneath it.
Notice what's missing There's no web.xml, no tomcat/ folder, and no XML configuration files anywhere. The server is embedded and configuration is automatic — that absence is Spring Boot's value, made visible.

The build file

pom.xml anatomy

The pom.xml (Project Object Model) is Maven's build file — it declares what your project depends on and how it's built. A Spring Boot pom has three parts worth knowing:

<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.x.x</version>
</parent>

<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>

<build><plugins>
    <plugin>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-maven-plugin</artifactId>
    </plugin>
</plugins></build>
  • parent — your project inherits from spring-boot-starter-parent, which supplies tested, compatible versions for every Spring library. This is why your dependencies need no <version> tags.
  • dependencies — the starters you've added. Each one is a single entry that pulls in many libraries behind the scenes.
  • plugin — the spring-boot-maven-plugin packages your compiled code, its dependencies, and the embedded server into the single runnable "fat" JAR.

Runtime configuration

application.properties

This file lives in src/main/resources and is the standard place to override Spring Boot's defaults. Spring Boot reads it automatically at startup — you never have to load it yourself. Each line is a key=value pair:

# Change the server port (default is 8080)
server.port=8081

# Name the application (appears in logs)
spring.application.name=springboot-demo

# Database connection (used in Session 2)
spring.datasource.url=jdbc:h2:mem:testdb

# Show generated SQL in the console
spring.jpa.show-sql=true

# Logging level
logging.level.org.springframework=INFO
  • You only add the keys you want to change — everything else keeps Boot's sensible defaults.
  • The same configuration can be written in YAML form as application.yml if you prefer its nested style.
  • Spring Boot also supports per-environment profiles (e.g. application-dev.properties) — useful later for separating dev and production settings.

For Lab 1 you leave this file empty — the defaults cover everything you need to run.

Save yourself some debugging

Common beginner mistakes

You'll likely hit one or two of these in the lab — and that's a good thing, because recognising them is part of learning. Here's what each looks like and why it happens:

  • Controller in the wrong package — if it's outside the main class's package, component scanning never finds it, so the bean is never created and the endpoint returns 404 Not Found with no obvious error.
  • Forgetting @Service / @RestController — without the annotation Spring doesn't manage the class, so any attempt to inject it fails at startup with "required a bean … that could not be found."
  • Port 8080 already in use — a previous run is still holding the port. Stop that process, or set server.port to a free value in application.properties.
  • Typo or wrong case in the URL — /Hello is not the same as /hello. URL paths are case-sensitive, so a mismatch gives a 404.
How to read errors When startup fails, scroll to the first red exception in the console — later errors are usually consequences of that one. The first line almost always names the real problem (a missing bean, a busy port, a syntax error).
