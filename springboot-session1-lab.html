Common issues

SymptomCauseFix
Startup fails with Port 8080 was already in use Another process (or a previous run) is holding port 8080. Stop the other process, OR add server.port=8081 to application.properties and re-run.
Maven can't download dependencies / build hangs No internet, or a corporate proxy is blocking Maven Central. Check your connection; configure the proxy in ~/.m2/settings.xml; then re-run the build.
IDE shows the project as plain files (not a Maven project) The project wasn't imported as Maven. Re-import: IntelliJ → right-click pom.xml → "Add as Maven Project"; Eclipse → right-click → Maven → "Update Project".
What just happened? You generated a fully-configured Spring Boot project, imported it, and started an embedded Tomcat server — with zero XML and no separate server install. The "Tomcat started on port 8080" line means your app is live and listening for requests.

LAB 2▶ Explore the Project Structure

No coding here — just open each file and read it. This builds familiarity with what Spring Boot generated.

  1. Open pom.xml. This file declares your project's dependencies and build configuration. Find three things:
    • the <parent> block → spring-boot-starter-parent (manages versions)
    • the <dependencies> section → spring-boot-starter-web
    • the <build> plugin → spring-boot-maven-plugin (builds the runnable JAR)
  2. Open the main class SpringbootDemoApplication.java. This is your application's entry point — the class that starts everything. Identify:
    • the @SpringBootApplication annotation on the class
    • the SpringApplication.run(...) call inside main
  3. Open src/main/resources/application.properties. This file holds your configurable application settings. It's currently empty — and that's perfectly fine. Spring Boot's defaults are doing the work.

Here is what the generated main class looks like — read it line by line:

package com.tcs.training;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class SpringbootDemoApplication {

    public static void main(String[] args) {
        SpringApplication.run(SpringbootDemoApplication.class, args);
    }
}
Why the package matters The main class sits in com.tcs.training. Because @SpringBootApplication triggers component scanning of this package and everything below it, your controllers and services must live in sub-packages like com.tcs.training.controller — otherwise Spring won't find them.

LAB 3▶ Your First REST Controller
Concept @RestController tells Spring this class handles web requests and returns the response body directly (not a view name). @GetMapping("/hello") maps HTTP GET requests for the /hello path to a method.
  1. Create a package com.tcs.training.controller.
  2. Create a class HelloController.java inside it.
  3. Add this code: 
    package com.tcs.training.controller;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HelloController {

    @GetMapping("/hello")
    public String sayHello() {
        return "Hello from Spring Boot!";
    }
}
  4. Run the app and open the endpoint in your browser.
Expected output Request: GET http://localhost:8080/hello
Response body (plain text): Hello from Spring Boot!
Try it yourself
  1. Change the returned string to your own message and re-run. Expected: the browser shows your new text.
  2. Add a second endpoint @GetMapping("/welcome") returning "Welcome!". Expected: http://localhost:8080/welcome shows Welcome!.
  3. Add a path that takes input: @GetMapping("/hello/{name}") with a @PathVariable String name parameter, returning "Hello, " + name. Expected: /hello/Purvam shows Hello, Purvam.
@GetMapping("/hello/{name}")
public String helloName(@PathVariable String name) {
    return "Hello, " + name + "!";
}
What happens under the hood Your browser sends GET /hello → the embedded Tomcat hands it to Spring's DispatcherServlet → which routes it to your sayHello() method → whose return value becomes the HTTP response body. That's the request flow from the slides, live.
LAB 4▶ Dependency Injection in Action
Concept @Service marks a class as a Spring-managed bean (a kind of @Component). @Autowired asks Spring to inject that bean where it's needed — so the controller never calls new on the service.
  1. Create a package com.tcs.training.service.
  2. Create GreetingService.java with a @Service annotation and a public method: 
    package com.tcs.training.service;

import org.springframework.stereotype.Service;

@Service
public class GreetingService {

    public String getGreeting() {
        return "Hello from the GreetingService bean!";
    }
}
  3. Inject it into HelloController with @Autowired and call it: 
    package com.tcs.training.controller;

import com.tcs.training.service.GreetingService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HelloController {

    @Autowired
    private GreetingService greetingService;

    @GetMapping("/hello")
    public String sayHello() {
        return greetingService.getGreeting();
    }
}
  4. Run the app and reload http://localhost:8080/hello.
Expected output GET http://localhost:8080/hello now returns: Hello from the GreetingService bean!
Observation question Remove the @Service annotation from GreetingService and re-run. What happens? Write down the behavior you see. (Expected: the app fails to start with an "unsatisfied dependency" / "required a bean of type GreetingService that could not be found" error — because without @Service, Spring no longer manages the bean to inject.)
The "aha" moment You never wrote new GreetingService(). Spring created the bean, stored it in its IoC container, and handed it to your controller. That's dependency injection — and it's what makes Spring apps so easy to test and change. Swap the service for a different implementation and the controller doesn't change at all.
Stretch goal Add a second method to GreetingService, e.g. getFarewell() returning "Goodbye!", and a new controller endpoint /bye that calls it. Expected: /bye returns Goodbye! using the same injected bean.
BONUS▶ Return JSON instead of plain text
Concept When a @RestController method returns an object (not a String), Spring Boot automatically serializes it to JSON using Jackson — which came in with the web starter. This is the first step toward building real APIs.
  1. Create a simple record in com.tcs.training.controller: 
    public record Greeting(String message, String from) { }
  2. Add an endpoint that returns it: 
    @GetMapping("/api/greeting")
public Greeting greetingJson() {
    return new Greeting("Hello from Spring Boot!", "GreetingService");
}
  3. Run and open http://localhost:8080/api/greeting.
Expected output (JSON) 
{
  "message": "Hello from Spring Boot!",
  "from": "GreetingService"
}

▶ Troubleshooting Guide

The most common things that trip up freshers in Session 1, and how to fix each one.

ProblemLikely causeFix
Endpoint returns 404 Not FoundController is outside the main class's package, or the path has a typoMove the controller under com.tcs.training.*; check the exact path and case
App fails: "Port 8080 was already in use"A previous run is still holding the portStop the old process or set server.port to a free port
"Field greetingService required a bean ... that could not be found"Missing @Service on the service classAdd @Service so Spring manages the bean
Red squiggles on @RestController / importsDependencies not downloaded yetRe-import / "Reload" the Maven project; check your internet
Changes don't show in the browserApp not restarted, or browser cached the pageStop & re-run the app; hard-refresh the browser (Ctrl/Cmd+Shift+R)
"Web server failed to start" with no clear reasonAnother error earlier in the logScroll up to the first red exception — that's the real cause
▶ Glossary — terms you'll hear today
TermPlain-English meaning
BeanAn object that Spring creates and manages for you
IoC containerThe part of Spring that builds beans and wires them together
Dependency Injection (DI)Giving an object the things it needs instead of it creating them
Auto-configurationSpring Boot setting things up automatically based on the classpath
StarterA bundle of related dependencies added with one line
Embedded serverA web server (Tomcat) packaged inside your app
EndpointA URL your app responds to (e.g. /hello)
DispatcherServletSpring's front controller that routes every request to the right method
POM"Project Object Model" — the pom.xml that defines your Maven build
ArtifactThe name/ID of your built project (here, springboot-demo)
